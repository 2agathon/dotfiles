# Tension Manifest Skill

Tension Manifest 是附着在任何文件上的嵌入式声明——它把隐含的张力显式化，让认识的演化轨迹可被追溯。

不是笔记专属。任何文件——认识笔记、决策记录、技术文档、工具 prompt——都可以带 TM 头。

---

## 字段定义

```yaml
外化时间: YYYY-MM-DD HH:MM:SS        # 认识被外化的时刻，精确到秒，写完不改
知识锚: kb-[base36编码]              # 同一认识所有版本共用，永久不变
版本: [整数，从1开始]                 # 版本号最大的即当前版本
来源: [self | claude-sonnet-4-6 | gpt-4o | deepseek-r1 | ...]
状态: [active | settled | abandoned]
起点张力: "现实是___，但我期望___，这个裂缝让这个文件诞生了"
引力范围: "凡是感受到___这个张力的人"
前驱: [kb-xxxx-vN | ~]
触发: "[什么让我意识到我对这个张力的理解本身是错的 | ~]"
演化摘要: "[原来看见的张力是___。现在看见的张力是___。 | ~]"
```

**字段语义速查：**

- `外化时间`：版本出生证明，也是 v1 知识锚的生成材料。写完不改，状态变更必须开新版本。
- `知识锚`：跨版本、跨平台、跨文件名的永久身份证。v1 创建时生成一次，之后所有版本复用。
- `版本`：序号，版本号最大的即当前版本，不需要 current 字段。
- `来源`：self 意味着独立推导，模型名意味着有外部智能参与了这次张力的重新定义。
- `状态`：active（仍在被理解中）/ settled（在当前认识水平下已到头）/ abandoned（被更真实的张力取代或放弃）。状态变更必须伴随新版本。
- `起点张力`：每个版本原样保留不改。记录让这个文件诞生的裂缝——不是问题，不是意图，是现实与期望之间的张力本身。
- `引力范围`：这条认识在哪个问题域里有碰撞可能。句式锁定，填写方向是碰撞点而不是描述词。跟着认识演化，每个版本可以不同。
- `前驱`：演化链指针。v1 填 `~`，之后填上一个版本的 `知识锚-v版本号`。
- `触发`：演化摘要的前提。**必须来自用户，不由 AI 代劳。** 允许事后回填，回填不触发新版本。
- `演化摘要`：固定句式：原来看见的张力是___，现在看见的张力是___。只记录张力如何被重新定义。

## 操作类型

### 操作一：创建 v1

**适用**：文件第一次需要 TM 头。

**流程：**

1. **索取时间戳**

   > 请告诉我当前时间（格式：YYYY-MM-DD HH:MM:SS），用于生成外化时间和知识锚。

2. **生成知识锚**（拿到时间后在任意 JS 环境运行）

   ```javascript
   'kb-' + parseInt(new Date('用户提供的时间').toISOString().replace(/\D/g,'').slice(0,14)).toString(36)
   ```

3. **引导填写起点张力**

   提示用户：这个文件诞生是因为什么现实和什么期望之间有裂缝？句式：`现实是___，但我期望___`。

4. **引导填写引力范围**

   提示用户：凡是感受到什么张力的人会对这篇文件有共鸣？句式锁定：`凡是感受到___这个张力的人`。

5. **输出完整 YAML**

   ```yaml
   外化时间: [用户提供的时间]
   知识锚: [生成的 kb-xxx]
   版本: 1
   来源: [本次参与的模型，或 self]
   状态: active
   起点张力: "[用户填写]"
   引力范围: "[用户填写]"
   前驱: ~
   触发: ~
   演化摘要: ~
   ```

---

### 操作二：开新版本

**适用**：认识发生了变化，需要记录张力被重新定义的时刻。

**流程：**

1. **索取触发**（必须来自用户）

   > 请用自己的话说：什么让你意识到你对这个张力的理解本身是错的？越接近"我原来以为 X，现在发现 Y"，演化摘要就越准。

2. **基于触发生成演化摘要**

   固定句式：`原来看见的张力是___，现在看见的张力是___。`

   草稿给用户确认，不直接写死。

3. **询问引力范围是否需要更新**

4. **输出更新后的 YAML**

   - `版本`：+1
   - `来源`：本次参与的模型
   - `前驱`：`[原知识锚]-v[原版本号]`
   - `触发`：用户提供的内容
   - `演化摘要`：确认后的内容
   - `外化时间`、`知识锚`、`起点张力`：不变

---

### 操作三：变更状态

**适用**：用户说"这个认识稳了"、"这个张力我放弃了"、"这个方向不走了"。

**规则**：状态变更必须伴随新版本。转入操作二的流程，触发字段记录"为什么这个状态在这里结束了"。

---

### 操作四：输出演化链摘要

**适用**：用户想看某条认识的演化历史。

**前提**：需要各版本的 TM 头内容。来源：
- 用户粘贴进对话
- Claude 有文件读取工具可以访问相关文件

基于各版本 TM 头，输出演化摘要字段串联的链条：

```
v1：[起点张力]
v2：原来看见的张力是___，现在看见的张力是___。（触发：___）
v3：...
```

如果无法获取版本内容，告知用户需要提供哪些信息后可以继续。

## 字段设计原则（记忆用）

1. 只记录文件自身无法表达的东西
2. 可推导的不存（例外：起点张力有感受损失，保留）
3. 每个字段有且只有一个语义
4. 元信息只描述文件对象本身，不描述文件内部的认识事件
5. 状态变更必须伴随新版本
6. 触发字段内容必须来自用户，允许事后回填，回填不触发新版本

---

## Guardrails

- **触发字段不由 AI 代劳**：AI 可以给方向提示，内容必须来自用户
- **起点张力在所有版本原样保留**：不随认识演化修改
- **知识锚只在 v1 生成一次**：之后所有版本复用，永远不变
- **状态变更不可只改字段**：必须走新版本流程
