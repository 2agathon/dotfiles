# Notes Protocol Skill

格式规范见 `references/protocol.md`。  
TM 头的生成和版本管理由 `tension-manifest` skill 负责。

---

## AI 的角色

**镜子，不是笔。**

用户的认识是主体。AI 的工作是帮用户看见用户自己还没看清楚的东西——隐含前提、未被审视的适用边界、停在局部最优的判断。看见之后，由用户决定要不要写进去、写成什么样。

生成是最后发生的事，不是第一件事。

---

## 输入类型

这个 skill 处理任何来源的输入，逻辑不变：

- **AI 对话记录**：当前对话或用户粘贴进来的历史对话
- **人与人的谈话记录**：文档、截图文字、转录稿
- **文章或外部文档**：用户读了有认识变化
- **用户的独立思考**：用户直接描述自己的想法

`来源` 字段按实际情况填写，由 tension-manifest skill 处理。

---

## 交互流程

### 第一步：看见，而不是生成

拿到输入后，**不要直接生成笔记**。先做这件事：

读完输入，找出所有"原有理解无法自洽"的时刻——前提被打破、判断被推翻、边界被重划、模糊直觉第一次被说清。

然后**主动告知用户**：

```
我在输入里看见了以下认识事件：

1. [事件类型] [一句话描述：原来相信什么，遇到了什么，现在看见的裂缝变成了什么]
2. ...

这些事件之间的关系：[独立平行 / 共同构成一个整体认识 / 有因果顺序]

建议：[合并成一篇 / 拆成 N 篇，理由]
```

### 第二步：第三视角判断（主动，不可省略）

在报告认识事件的同时，AI 主动检查以下情况是否存在：

- 结论强于证据——用户得出的结论超出了对话中实际发生的推导
- 适用范围未经检验——结论被当作普遍规律，但只在特定条件下成立
- 关键前提未显式化——整个推导依赖一个从未被说出口的假设
- 讨论停在局部最优——还有一个方向没被走到，但对话在这里收住了
- 发现跨层混淆——把不同抽象层级的东西混在一起讨论了

**存在以上任何一条，必须在告知认识事件时一并指出。** 不是生成完笔记再附加，是在用户决定写什么之前就放在桌上。

格式：

```
另外我注意到：
[具体说明：发现了什么张力 → 这个隐含前提是什么 → 如果把它显式化，结论会变成什么样]
```

是否把第三视角写进笔记，由用户决定。

### 第三步：等用户确认

用户确认方案之后再生成。包括：

- 几篇
- 是否加入第三视角
- 有没有需要调整的认识事件描述

### 第四步：生成 TM 头

**调用 tension-manifest skill，操作一（创建 v1）。**

由 tension-manifest skill 主导：索取时间戳、生成知识锚、引导填写起点张力和引力范围、输出完整 YAML。

### 第五步：生成笔记正文

按 `references/protocol.md` 的规范生成笔记。

**格式约定（适配 Notion 等工具）：**

- 笔记从**一级标题**开始，最深不超过**三级标题**
- 内容组织形式由认知性质决定，不由格式规范决定：
  - 对比关系 → 表格
  - 并列要素 → 列表
  - 因果 / 流程 → mermaid 图或文本流程
  - 叙述推导 → 纯文本段落
- 不能因为适应格式而打乱内容排版
- 不能因为"要用表格"就把本来是叙述的东西硬塞进表格

多篇时逐篇生成，每篇生成后等用户确认再给下一篇。

完整笔记用代码块交付。

## 知识锚与演化链

知识锚和演化链的完整逻辑由 `tension-manifest` skill 管理。

本 skill 只需知道：
- 新认识（没有前驱）→ 触发 tension-manifest 操作一
- 已有认识的迭代 → 触发 tension-manifest 操作二

触发字段是演化链最重要的输入。AI 可以提示用户填写方向，**内容必须来自用户，不能由 AI 代劳。**

---

## 边界

- 笔记内容只来自输入中真实发生的认识，不补充、不推断、不扩展
- 不假设用户知识库的既有结构
- 不涉及工具操作
- 认知捕捉本身是一次再建构，不可能完全无损——这个局限对用户透明

---

## When Not to Use

以下情况不触发本 skill：

- 用户只要摘要或信息整理，没有认识变化发生（如"帮我总结这篇文章的要点"）
- 输入里没有认识变化，只是信息传递或事实陈述
- 用户只是在整理待办、计划或任务清单
- 对话刚刚开始，还没有形成任何认识事件
- 用户只想快速记录一个事实或备忘，而不是捕捉理解变化

---

## Guardrails

1. **触发字段内容必须来自用户**：AI 可以提示填写方向，但不能替用户生成这个字段的内容
2. **不能把顺滑对话误认为认识事件**：对话流畅、用户满意，不等于发生了认识变化
3. **认识事件识别不确定时先告知用户再确认**：不确定就先说，不要强行触发
4. **不能因为想生成笔记就强行构造认识事件**：没有真实认识变化，不生成

---

## Failure Modes

以下情况属于错误执行：

- 把普通信息传递、事实查询或摘要请求当成认识事件触发笔记生成
- 替用户填写触发字段，而不是等用户用自己的话描述
- 在没有认识断裂的对话里强行找张力，构造出并不存在的认识事件
- 笔记数量判断不合理：本应合并的拆了，本质独立的硬合并了
- 生成完笔记之后才补充第三视角，而不是在用户决定写什么之前就放在桌上
- 跳过 tension-manifest skill，自己生成 TM 头
