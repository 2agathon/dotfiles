# Knowledge Shaping

## Purpose

把雾状知识输入塑形成稳定知识对象。

这里的“稳定”不是最终完成，而是已经具备最小可调用性：未来可以被复用、比较、补写、重读，而不必重新回到原材料重新捞一遍。

本 Skill 处理的是这样一类输入：

- 有知识增量
- 但尚未形成明显的认识断裂
- 尚未进入可承诺的行动规格
- 尚未形成需要沉淀的决策落点

它不是摘要原文，不是表达感想，不是生成规格，也不是记录决策。

---

## When to Use

在以下情况触发：

- 看完一篇材料，对某个对象有了初步把握
- 学到一个概念，但还没到认识断裂的程度
- 看清一个机制，但还不需要转成行动规格
- 分清两个常混对象的差别
- 发现某个框架常被越界使用，需要先划边界
- 有东西值得留下，但还不该被推进成 notes/spec/decision
- 还说不清对象，但已经感觉到这里存在一个值得塑形的知识轮廓

---

## When Not to Use

以下情况不应优先触发：

- 已经发生明显认识断裂，应优先进入 `notes-protocol`
- 已经需要形成行动承诺，应优先进入 `conversation-to-spec`
- 已经形成明确决策落点，应优先进入 `decision-record`
- 用户只是要摘要原文，不需要对象塑形
- 输入仍然过于空泛，连暂时对象都无法显影

---

## Core Boundary

### 与 `notes-protocol` 的边界

- `notes-protocol` 记录认识如何变化，为什么变化，旧理解如何失效，张力如何演进
- `knowledge-shaping` 只处理当前这份输入里，可稳定保留的知识对象是什么

可以这样区分：

- `notes-protocol` 关心“我为什么变了”
- `knowledge-shaping` 关心“我现在得到的对象是什么”

### 与 `conversation-to-spec` 的边界

- `conversation-to-spec` 把讨论压成承诺、规格、实验或工作语言冻结
- `knowledge-shaping` 不构造承诺，只塑形知识，不把知识推进成行动

### 与 `decision-record` 的边界

- `decision-record` 记录选择的落点、背景、取舍与重评条件
- `knowledge-shaping` 不记录“选了什么”，只记录“当前知道了什么”

### 与 `assumption-audit` 的边界

- `assumption-audit` 的主任务是识别、分类、审计隐含前提
- `knowledge-shaping` 可以写边界、常见误读与不确定处，但不以“审计前提”为主线

---

## Core Principles

### 1. 先塑对象，不先追深刻

不要把输入一上来拔高成洞察。  
先问：这里最值得稳定保留的知识对象是什么。

### 2. 从定义推导结构，不靠经验枚举模板

对象结构应从知识对象的最小维度出发：

- 结构：它是什么
- 动态：它怎么运作
- 边界：它不是什么，在哪里失效，容易和什么混

不要先从“大家通常会记哪些卡片”出发。

### 3. 边界不是附属信息，而是对象的一部分

很多知识不是死在“定义不够”，而是死在“边界没写”。  
所以边界不能推迟，不能可有可无，不能只在高级版本里补。

### 4. 可以不完整，但不能装完整

允许对象目前只看清一部分。  
但必须明确哪里还不稳、哪里还模糊，而不是把不清楚写成完整。

### 5. 输出必须可被未来调用

结果不能只是一次性阅读产物。  
它必须能够被后续比较、补写、重读、引用。

### 6. 对象可以从模糊开始，但不能停留在模糊感里

如果对象一开始不清楚，不应立即停止。  
应先做对象显影，把“这里有个东西”的感觉压成一个暂时对象，再继续塑形。

---

## Object Model

每个知识对象都要尽可能回答三类问题：

### 结构维

- 它是什么
- 它由什么构成
- 它和相邻对象的最小区别是什么

### 动态维

- 它怎么运作
- 它在什么条件下成立
- 它如何变化、如何起作用、如何失效

### 边界维

- 它不是什么
- 它在哪里不适用
- 它最容易和什么混
- 它一旦越界，会出现什么错误理解

这三维不是三种可选卡片，而是一个知识对象的最小成立条件。

---

## Primary Shapes

在实际输出时，可以根据输入重心，让对象以不同形态呈现。  
这些形态只是重心不同，不是根本类型不同。

### 概念型

当输入主要在回答“它是什么”。

适合：

- 术语
- 基本概念
- 定义性对象

输出时，结构维为主，动态维和边界维不能缺失，但可以较短。

### 机制型

当输入主要在回答“它怎么运作”。

适合：

- 机制
- 流程
- 因果链
- 行为模式

输出时，动态维为主，结构维和边界维仍必须在场。

### 对比型

当输入主要在回答“它和什么不同”。

适合：

- 易混概念
- 相邻方案
- 同类框架的区分

输出时，对比关系是主线，但仍要落回结构维、动态维、边界维。

---

## Workflow

### Step 1. 识别或显影知识中心

先判断输入里是否已经存在一个可明确命名的中心对象。

如果有，直接进入后续塑形。

如果没有，不停止，而是先做 **对象显影**。对象显影要做的不是勉强下定义，而是把“感觉这里有个东西”压成一个暂时对象。至少要回答：

- 当前感觉“有东西”的部分是什么
- 它更像概念、机制，还是对比
- 它目前最粗的轮廓是什么
- 哪些地方已经能说，哪些地方还不能说

对象显影的目标不是一次完成对象，而是让它从雾感进入最小可塑状态。

如果连暂时对象都无法形成，才停止，并明确说明当前输入仍不足以进入 `knowledge-shaping`。

### Step 2. 提取三维内容

围绕中心对象或暂时对象，分别抽取：

- 结构维内容
- 动态维内容
- 边界维内容

允许其中某一维较弱，但不能完全缺席。  
如果某一维拿不到内容，要显式写出“当前不足”。

### Step 3. 压缩为对象，而不是复述材料

删掉不服务对象成形的背景信息。  
输出的中心应该是对象本身，而不是材料叙述顺序。

不要按“作者先说了 A，再说了 B”来写。  
要按“对象目前成了什么样”来写。

### Step 4. 显式写出边界与误读

这一层必须单独写，不可省略。至少要处理：

- 它不等于什么
- 最容易和什么混
- 越界时会怎样
- 当前最常见的误读是什么

### Step 5. 标出未完成处

如果对象还不稳定，必须明确指出：

- 哪些地方只是暂时成形
- 哪些关系仍模糊
- 哪些判断仍需补材料

这一步不是可选谦虚，而是对象诚实度的一部分。

### Step 6. 结束于对象本身

流程到这里就结束。  
`knowledge-shaping` 不以“它接下来要去哪”作为完成条件。

对象可以先停在这里。  
是否被别的 Skill 继续使用，是后续动作，不是本 Skill 的内建压力。

## Quality Bar

一个合格的 `knowledge-shaping` 输出，应至少满足以下条件：

### 1. 能恢复对象

未来重读时，不必回原材料，也能较快恢复这个对象。

### 2. 能分清邻近对象

输出必须让对象与相邻对象区分开，而不是越写越糊。

### 3. 不伪装成洞察

它可以清楚，但不必深刻。
不能把普通知识整理伪装成重大认识事件。

### 4. 边界明确

对象必须写出不适用处、易混处或越界征兆。
如果没有边界，这个对象就还没成形。

### 5. 对不确定处诚实

不知道就是不知道。
不能把暂时性理解写成稳定结构。

### 6. 能接住模糊起点

对象不必一开始就清楚。
但输出必须显示：它已经从“雾感”推进到“最小对象”。

---

## Failure Modes

### 1. 退化成摘要

只是在复述材料顺序，没有形成对象。

### 2. 退化成感想

只表达态度，没有形成可调用对象。

### 3. 退化成伪洞察

把常规整理包装成高深认识。

### 4. 只有定义，没有边界

看似清楚，实际什么都能套。

### 5. 只有区分，没有对象

写了很多“它不是什么”，但始终没写清“它是什么”。

### 6. 只有机制，没有结构

过程讲得很多，但对象本身是谁不明确。

### 7. 把雾状输入当作输入不足

最需要被接住的，恰恰是“感觉这里有个东西，但没说清楚”的状态。
如果一遇到模糊就停止，这个 Skill 就失去存在意义。

### 8. 把可选附注重新写成强制路由

`knowledge-shaping` 的完成条件是对象成形，不是对象被送去别处。
不要把“去向”重新变成内建压力。

## Interfaces

### 与 `drift-detector`

同一知识对象被反复修改时，`drift-detector` 可以检查：

* 结构是否漂移
* 动态理解是否变化
* 边界是否收缩或扩张

### 与 `role-lens`

`role-lens` 的前提，是先有一个相对稳定的对象可供重读。
`knowledge-shaping` 产出的对象，就是它的输入基础。

### 与 `notes-protocol`

如果某个对象后来引发认识断裂，它可以作为“断裂前对象”进入 `notes-protocol`。

### 与 `conversation-to-spec`

如果某个对象后来开始支撑行动承诺，它可以成为 spec 的背景知识对象。
但这不是本 Skill 的完成要求。

### 与 `decision-record`

如果某个对象后来成为决策背景，它可以被引用。
但 `knowledge-shaping` 本身不负责推动这一跳。

---

## Scope for v0.1

初版直接支持三种主形态：

* 概念型
* 机制型
* 对比型

但无论哪一种，三维结构都必须完整在场：

* 结构维
* 动态维
* 边界维

并且初版就必须支持一种能力：

* **对象显影**

也就是允许输入从“对象尚未清楚”的状态进入塑形。

初版不做：

* 额外扩展更多对象种类
* 复杂路由规则
* 自动升级机制

先把一件事做好：

**把普通知识输入，稳定地塑形成对象。**

---

## Success Condition

`knowledge-shaping` v0.1 成功，不看它写得多漂亮，只看四件事：

1. 能接住普通输入
2. 能接住模糊输入
3. 能产出稳定对象
4. 能和现有 Skill 保持边界

---

## One-line Definition

`knowledge-shaping` 是一个把雾状知识输入塑形成稳定知识对象的 Skill，用于承接尚未成为认识事件、行动规格或决策落点的中间层材料。
