## 这个 skill 做什么

在动手之前，输出一份设计文档。
对齐目标、暴露假设、发现边界问题——在动手之前，不是动手之后。

---

## 触发规则

**默认触发：**

- 用户描述了目标，但没有说边界
- 任务涉及新的结构（目录、文件、skill、文档体系）
- 任务涉及多个区域联动
- 用户说"就改一下"——**默认触发，除非用户明确说不需要**

**用户侧触发权：**
用户任何时候可以主动说"先做个 plan"，AI 必须执行，不需要判断是否必要。

**断点续传：**
用户说"继续 {名称}"→ 从 `.ai/plans/` 读取。说"继续"但没指定 → 列出进行中的 plan 供选择。

**判断规则：** 宁可多触发被用户说不需要，也不跳过导致方向跑偏。

---

## When Not to Use

- 用户明确说不需要 plan
- 任务边界极其清晰，只有一种合理路径，无需对齐目标
- 修复明确的单点问题，改动范围完全确定

---

## 强制停止点

以下必须停下来等用户确认——不能 ASSUMPTION 绕过：

- **目标不清楚**：只问这一个问题，不展开其他
- **"不覆盖什么"用户没有说**：必须问，不能自行推断后填空
- **Vibe Plan 输出后**：用户确认之前不动手执行——**没有例外**

> "以上是我的理解，有没有偏差？确认后开始第一步。"

---

## 信息收集

**必须来自用户：**

- 任务的目标（做完之后什么变了）
- 明确不覆盖什么

**可以从上下文推断，用 `[ASSUMPTION]` 标注：**

- 涉及的区域
- 约束的细节

**信息缺口处理：**

| 情况           | 处理方式                                |
| -------------- | --------------------------------------- |
| 目标不清楚     | 只问这一个问题，其余占位                |
| 不覆盖什么没说 | 必须问，不能推断                        |
| 涉及区域不确定 | `[ASSUMPTION]` 标注，在 plan 里显式列出 |
| 约束不知道     | `[OPEN QUESTION]` 标注                  |
| 其余缺口       | 只问最影响方向的一个问题                |

---

## 输出结构

在对话中展示给用户确认：

```
## Vibe Plan · [名称]

### 目标
做完之后什么变了。

### 不覆盖
明确不处理什么。

### 工作区域与边界
涉及哪些区域，各自职责，边界在哪里。

### 关键决策
核心选择，每个说明选项和权衡。
[OPEN QUESTION] 标注不确定的。

### 假设
[ASSUMPTION] 列出所有推断。

### 执行顺序
分步骤，每步是可独立验证的交付单元。
每步再拆成具体 task——AI 单次交互能高质量完成的粒度。

步骤 1：{交付单元}
  - Task 1.1：{做什么，改哪里，预期结果}
  - Task 1.2：...
步骤 2：...
```

### 代码场景补充

**工作区域与边界**：用目录或文件名说明，不只是文字描述。

**Task 粒度**：一个 task 只改一个关注点（一个函数、一个接口、一个组件），输出可独立验证。超过三句话描述 → 再拆。

```
步骤 1：建立数据模型
  - Task 1.1：创建 User model（字段：id, email, role），验证：migration 跑通
  - Task 1.2：创建 Session model（字段：id, userId, expiresAt），验证：关联查询可用
```

### 通用场景补充

**工作区域与边界**：用具体边界名称说明——skill 名、文档章节、知识域、系统模块。

**Task 粒度**：一个 task 只处理一个关注点（一个 skill、一个文档章节、一个决策点），输出可独立验证。

```
步骤 1：定义治理声明格式
  - Task 1.1：在 governance-protocol.md 写出字段定义，验证：能覆盖 bail-out 和 code-modify 两个样本
  - Task 1.2：给 bail-out 加声明，验证：声明与 skill 实际行为一致
```

## 保存到文件

### 是否保存

用户确认 plan 后，AI 问一句："需要保存到文件吗？"
用户说保存 → 写文件。用户说不用 → 不写。

### 何时写文件

| 时机 | 行为 |
|------|------|
| plan 在对话中输出 | 不写文件，等用户确认 |
| 用户确认 plan，且确认保存 | 写文件到 `.ai/plans/{名称}.md`，目录不存在则创建 |
| 该 plan 文件已存在 | 问用户：覆盖还是新建 |

### 文件格式

用户确认后，写入的文件在对话输出的基础上，给每个 task 加状态字段：

```markdown
# Plan · {名称}
创建时间：{时间}
状态：进行中

## 目标
{同 plan}

## 不覆盖
{同 plan}

## 工作区域与边界
{同 plan}

## 关键决策
{同 plan}

## 假设
{同 plan}

---

## 执行

### Step 1：{交付单元名}

#### Task 1.1：{具体做什么}
状态：⬚ 未开始

#### Task 1.2：{具体做什么}
状态：⬚ 未开始

### Step 2：{交付单元名}

#### Task 2.1：{具体做什么}
状态：⬚ 未开始

---

## 执行日志
```

### 执行过程中更新文件

| 时机 | 行为 | 写文件范围 |
|------|------|-----------|
| task 完成 | 更新该 task 的状态行和备注 | **只改该 task 的状态/改动/备注行** |
| 需要加 task 或调整顺序 | 直接改执行段，标注 `[MODIFIED: 原因]` | 只改执行段 |
| 需要改目标/边界/设计 | 停下来等用户确认 | 确认后才改设计段 |
| 收工 | 追加收工摘要到"执行日志"段 | 只在文件末尾追加 |
| 全部 task 完成 | 文件顶部状态改为"已完成" | 只改状态行 |
| plan 被放弃 | 文件顶部状态改为"已放弃" | 只改状态行 |

**关键约束：更新 task 状态时，严禁重写整个文件。只定位到对应 task 的状态行，做局部修改。**

task 完成后的状态格式：

```markdown
#### Task 1.1：{具体做什么}
状态：✅ 完成
改动：{涉及的文件或区域}
备注：{发现或备注，无则不写}
```

task 卡住时的状态格式：

```markdown
#### Task 2.1：{具体做什么}
状态：🔄 进行中
卡点：{卡在哪}
```

### 新会话接手

用户说"继续 {名称}"时：
1. 读 `.ai/plans/` 下对应文件
2. 找到第一个非 ✅ 的 task，从那里继续
3. 如果有卡点描述，先处理卡点

用户说"继续"但没说哪个时：
1. 读 `.ai/plans/` 目录，列出所有状态为"进行中"的 plan
2. 问用户继续哪个

## 这个结构在保护什么

保护 AI 协作的可控性。AI 生成速度快，方向一旦偏了修正成本极高。
Vibe Plan 是强制减速点——不是为了完美设计，是为了在动手之前暴露分歧。

**不走 Vibe Plan 会发生什么**
AI 从头做到尾交付一整块成果，里面有五个需要讨论的决策但已经全部落地了。用户发现方向不对，推倒重来成本太高，只能将就。

---

## 边界

这个 skill 不处理执行细节——plan 确认后，执行由用户逐步推进。
Vibe Plan 不是完美设计文档，是对齐工具。发现新信息时随时更新，不需要重走完整流程。

---

## 自检（AI 执行）

- [ ] "不覆盖"有没有明确写？空着 → 停下来问用户。
- [ ] 执行顺序里每一步是否可以独立验证？
- [ ] 每个 task 是否是 AI 单次能完成的粒度？超过三句话 → 需要再拆。
- [ ] 有没有在用户确认前就动手执行？有 → 立刻停止。
- [ ] 有没有在用户确认前就写了 plan 文件？有 → 删掉，等确认。
- [ ] 目标和不覆盖的答案是来自用户，还是 AI 推断的？推断的 → 停下来确认。
- [ ] 更新 task 状态时，是否只改了状态行？重写整个文件 → 错误执行。

---

## Failure Modes

以下属于错误执行：

- 在用户确认 Vibe Plan 之前就动手执行
- 在用户确认 Vibe Plan 之前就写了 plan 文件
- 目标和"不覆盖"的内容来自 AI 推断，而不是用户明确说明
- 执行顺序里的步骤无法独立验证
- Plan 过于抽象，只有方向描述，没有具体到可以指导执行的程度
- 跳过 plan 直接进入执行，理由是"任务很简单"——简单性判断必须来自用户
- 执行顺序只有步骤没有 task，粒度太粗
- 更新 task 状态时重写了整个 plan 文件
- 需要改目标/边界/设计时没有停下来等用户确认
