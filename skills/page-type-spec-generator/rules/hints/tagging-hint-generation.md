# tagging-hint 生成规则

## 目标对象

生成 `tagging-hint.md`。

## 输入

1. `反例` 中会直接改变段打标边界裁决的内容
2. `采集和转化规范` 中会直接改变段打标命中、排除或局部规范化的规则
3. 当前 type 的标签集合

## Stage 6：Skeleton

输出面向下游 LLM 的紧凑骨架：

```md
# {类型名} · 段打标专项规则

## 标签边界裁决

- __PTSG_PENDING__

## 局部规范化与排除

- __PTSG_PENDING__

## 特殊段与版式重复段

- __PTSG_PENDING__

## OCR / 排版噪声

- __PTSG_PENDING__

## 优先级补充

- __PTSG_PENDING__
```

规则：

1. 每条规则优先写成 `A vs B：...`、`若 ... 则 ...`、`出现 ... 时 ...` 这类短硬格式。
2. 若某节对该 type 确无专项增量，可删除该节；不要写“无特殊规则”“遵循系统默认”等填充句。
3. 若存在应有专项规则但来源不稳，保留 `__PTSG_PENDING__`。
4. 每条规则应尽量保持单行或紧凑短句，不展开背景解释。

## Stage 7：Semantic Draft

1. 只写段打标阶段的页类型专项增量。
2. 优先输出冲突对、冲突簇、标签家族边界；不要退回“一标签一句”提醒。
3. `标签边界裁决` 只写会改变命中归属的边界规则，例如哪个标签优先、哪个表达归谁、不应误判成谁。
4. `局部规范化与排除` 只写当前段内即可执行、且会直接影响命中或 `tag_value` 的规则；不要写字段来源说明。
5. `特殊段与版式重复段` 只有在当前 type 确有专项处理规则时才写；没有就删掉该节。
6. `OCR / 排版噪声` 只写会影响打标的页类型特有噪声，不写通用 OCR 常识。
7. `优先级补充` 只写多标签竞争、多候选竞争或局部歧义时真正决定取舍的补充规则。
8. 若某条规则已经能从 `value_hint`、`context_hint` 或标签定义直接得到，不要再写进来。
9. 最终内容应让下游模型直接决定：标谁、不标谁、冲突时谁优先、值如何局部规范化。
10. 优先使用短句、并列短语、条件句或冒号句；不要写成长段说明文。

## 禁止

1. 不得把同一句泛化提醒换标签名后批量罗列。
2. 不得把 `value_hint`、`context_hint` 或标签定义按标签重复抄进来。
3. 不得写块组装、块摘要、页摘要策略。
4. 不得写模型行为解释、消费方背景、系统机制说明或其他元解释噪音。
5. 不得把 `局部规范化与排除` 写成取值流程、数据血缘或字段来源说明。
6. 不得把 `优先级补充` 写成抽象口号，例如“结合上下文综合判断”。
7. 不得写“该类型常见于……”“该规则用于帮助模型理解……”这类说明文句子。

## 失败信号

1. Semantic Draft 仍主要是一标签一句模板提醒。
2. 规则里大部分只是“未提及 X 不要标注 X”。
3. 文件没有真正的标签对边界，只有泛化套话。
4. 大部分规则只是把单标签 `反例` 改写成排除句，没有形成标签家族边界。
5. 规则主体仍然围绕单个标签逐条展开，而不是围绕冲突对、冲突簇或标签家族展开。
6. 读完后仍不能直接裁决“标谁 / 不标谁 / 谁优先 / 如何局部规范化”。
7. 各小节存在，但内容仍是说明文，不是 `模式 -> 决策` 的执行句。
