# assembly-hint 生成规则

## 目标对象

生成 `assembly-hint.md`。

## 输入

1. 当前 type 的 block 集合
2. `采集和转化规范` 中会直接改变块组装结果的规则
3. `缺项处理` 中会直接改变成块 / 抑制成块判断的规则
4. `冲突处理` 中会直接改变块级取舍顺序的规则

## Stage 6：Skeleton

输出按 block 决策展开的骨架：

```md
# {类型名} · 块组装专项规则

## 全局组装规则

- __PTSG_PENDING__

## {Block 名称}

成块前提：__PTSG_PENDING__
抑制条件：__PTSG_PENDING__
归并与拆分：__PTSG_PENDING__
可补挂标签：__PTSG_PENDING__
记录时间取值：__PTSG_PENDING__
冲突取舍：__PTSG_PENDING__

## {下一个 Block 名称}

成块前提：__PTSG_PENDING__
抑制条件：__PTSG_PENDING__
归并与拆分：__PTSG_PENDING__
可补挂标签：__PTSG_PENDING__
记录时间取值：__PTSG_PENDING__
冲突取舍：__PTSG_PENDING__
```

规则：

1. 每个 block 都必须有自己的小节，不得把所有规则压成 type 级散文。
2. `全局组装规则` 只写会同时影响多个 block 的规则；单个 block 的规则写回该 block 小节。
3. 若某个字段对该 block 确无专项增量，可删除该行；不要写“无特殊规则”“遵循系统默认”等填充句。
4. 若存在应有专项规则但来源不稳，保留 `__PTSG_PENDING__`。
5. 每个字段都应保持短、硬、可执行；优先单行或紧凑短句。

## Stage 7：Semantic Draft

1. 只写块级决策，不写 tag 级取值定义。
2. 只保留会直接改变块组装结果的规则。
3. `成块前提` 只写这个 block 何时应成立、哪些证据组合足以成块。
4. `抑制条件` 只写缺哪些关键证据时应跳过、抑制或不输出该 block。
5. `归并与拆分` 只写会改变块数量或块边界的规则，例如并块、拆块、窗口拆分、对象归并。
6. `可补挂标签` 只写会直接改变块内容或归属结果的补挂规则；不要退回单 tag 取值说明。
7. `记录时间取值` 只写当前 type / 当前 block 的专项口径；不要重复系统默认规则。
8. `冲突取舍` 只写真正决定块级优先顺序、重叠证据处理、相近 block 归属的规则。
9. 若某条规则不会改变“保留 / 跳过 / 归并 / 拆分 / 补挂 / 取记录时间 / 冲突裁决”的结果，就不要写进来。
10. 若 block description 已经表达块语义，不要在这里改写一遍；这里只写决策补丁。
11. 最终内容应让下游模型直接决定：这个 block 该不该出、怎么出、出几个、缺项时怎么办、冲突时保留谁。
12. 优先使用短句、并列短语、条件句或冒号句；不要写成长段说明文。

## 禁止

1. 不得写单 tag 的取值示例或识别示例。
2. 不得把 `value_hint` / `context_hint` 内容搬进来。
3. 不得写块摘要或页摘要策略。
4. 不得把 `全局组装规则` 写成系统机制说明、数据结构解释或其他元解释噪音。
5. 不得把 block description 改写后充当组装规则。
6. 不得围绕单个 tag 名称长篇展开，除非该 tag 会直接改变成块、抑制、补挂或拆分决策。
7. 不得把 `记录时间取值` 写成默认流程复述。

## 失败信号

1. 最终内容主要是 tag 级示例，而不是 block 级决策。
2. 规则读完后不能直接指导“保留/跳过/归并/拆分”决策。
3. 文件里只剩下泛化空话，没有真正改变组装结果的规则。
4. 规则主体围绕单个 tag 名称展开，而不是围绕 block 或块级条件展开。
5. 各 block 小节存在，但读完后仍不知道该 block 何时成块、何时抑制、何时拆分或并块。
6. 文件主要在解释系统如何工作，而不是给下游模型直接可执行的块级决策规则。
