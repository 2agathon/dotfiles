# assembly-hint 生成规则

## 目标

生成 `assembly-hint.md`。

这个文件会直接插入块组装 prompt，主要消费方是：

1. `src/module/material/ai/block_ai.py`
2. `src/module/material/ai/prompts.py`

系统 prompt 已经定义了块组装的基本算法；`assembly-hint.md` 只补充该页类型特有的组装决策。

## 输入

1. `采集和转化规范`
2. `缺项处理`
3. `冲突处理`

## 总规则

1. 只写块级决策，不写 tag 级取值定义。
2. 只写 page-type-specific 的例外和取舍，不重复系统 prompt 已固定的步骤。
3. 每条规则都应能直接改变块组装结果。

## Stage 6：Skeleton

输出固定骨架：

```md
# {类型名} · 块组装处理指导

## 采集与规范化

__PTSG_PENDING__

## 缺项处理

__PTSG_PENDING__

## 冲突处理

__PTSG_PENDING__
```

## Stage 7：Semantic Draft

### 采集与规范化

1. 只保留跨 tag 或块级可执行的规则。
2. 适合这里的内容包括：
   1. 同类标签归并
   2. 正文与表格并存时如何纳入
   3. 拆块或并块的业务口径

### 缺项处理

1. 明确“缺什么时仍可成块，缺什么时应抑制成块”。
2. 若规则只影响某个 block，应显式点名 block。

### 冲突处理

1. 只写真正决定取舍顺序的规则。
2. 例如：时间锚点优先、正文优先于页眉、主诊断优先于次诊断。

## 写法要求

1. 每条规则尽量短、可执行、可直接映射为组装决策。
2. 优先用“当……时，……”句式表达条件和动作。
3. 若规则只对某一类 block 生效，必须写明对象。

## 禁止

1. 不得写单 tag 的取值示例或识别示例。
2. 不得把 `value_hint` / `context_hint` 内容搬进来。
3. 不得写块摘要或页摘要策略。

## 失败信号

1. 最终内容主要是 tag 级示例，而不是 block 级决策。
2. 规则读完后不能直接指导“保留/跳过/归并/拆分”决策。
3. 文件里只剩下泛化空话，没有真正改变组装结果的规则。
