# 交互点协议

## 总规则

1. 只在真正需要用户决定的边界点交互。
2. 机械步骤不得打断用户。
3. 可自动继续的问题应记录到报告，而不是默认升级为交互。
4. 不得在前置条件未满足时提前发问。

## 运行模式

### `interactive`

1. 允许执行全部硬交互点。
2. 允许执行必要的软交互点。

### `constrained`

1. 只保留硬交互点。
2. 软交互点默认自动继续并记录。

### `no_interaction`

1. 不执行任何交互点。
2. 所有原本需要交互的问题都必须显式记录到最终报告。

## 硬交互点

以下情况必须请求用户决定，除非当前模式为 `no_interaction`：

### 1. 选择要处理的 `type`

触发条件：

1. `identity-map.json` 已枚举出一个以上 `type`

动作：

1. 列出检测到的 `type`
2. 让用户选择本次处理范围

禁止：

1. 不得在 `identity-map.json` 生成前询问该问题
2. 不得在可交互模式下默认处理全部 `type`

### 2. 高冲突身份判定

触发条件：

1. 空 `tag编码` 行存在多个同样合理的身份解释
2. 多个 canonical id 都可成立，且无法按规则唯一收敛

动作：

1. 列出冲突对象
2. 列出候选解释
3. 让用户选择

### 3. 已有 taxonomy 冲突

触发条件：

1. 用户提供了 `page-types.data.json` 和 `page-types.tree.json`
2. 本次结果与已有 type / category 出现真实冲突

动作：

1. 列出冲突位置
2. 让用户决定追加、覆盖、跳过或停止

### 4. block 边界歧义

触发条件：

1. 某行既可能归属当前 block，也可能应独立成新 block

动作：

1. 列出两种解释
2. 让用户选择

## 软交互点

以下情况默认可自动继续；仅在 `interactive` 模式下，且确有必要时才可询问用户：

### 1. 表头 typo 修复

默认动作：

1. 自动修复
2. 记录到 `normalization-report.md` 和 `audit-report.md`

### 2. 小范围文本规范化

默认动作：

1. 自动清洗
2. 记录到 `normalization-report.md`

### 3. `anchor_binding` 轻语法修复

默认动作：

1. 自动修复
2. 记录到 `audit-report.md`

### 4. 素材不足时输出空模板

默认动作：

1. 不打断用户
2. 输出空模板
3. 记录到 `final-report.md`

## 禁止打断的情况

以下情况不得触发交互：

1. `run-manifest.json` 生成
2. `workbook.raw.json` 生成
3. `workbook.normalized.json` 生成
4. `identity-map.json` 中无冲突的正常身份抽取
5. `skeleton/` 生成
6. `schema` 结构校验
7. 统计类信息汇总

## 报告要求

如果某个问题按规则自动继续，必须在报告中写明：

1. 问题是什么
2. 自动采取了什么动作
3. 是否仍有剩余风险

## 失败信号

以下情况属于交互协议错误：

1. 在类型枚举前就要求用户选择 `type`
2. 把机械修复升级成硬交互点
3. 在 `no_interaction` 模式下仍尝试等待用户回答
4. 自动继续后没有留下记录
