# value_hint 生成规则

## 目标对象

生成 `tags[].value_hint`。

## 输入

1. `tag定义`
2. `触发模式`
3. `正例`
4. `反例`

## Stage 6：Skeleton

固定输出以下四行：

```text
定义：{tag定义或 __PTSG_PENDING__}
触发：{触发模式或 __PTSG_PENDING__}
正例：{正例或 __PTSG_PENDING__}
反例：{反例或 __PTSG_PENDING__}
```

规则：

1. 四个槽位必须全部显式出现。
2. 某列为空时必须填 `__PTSG_PENDING__`。
3. Skeleton 阶段不得把正例和反例抽象成规律句。

## Stage 7：Semantic Draft

1. 输出 2-5 句自然语言短段落。
2. 建议顺序：
   1. 抓取对象
   2. 主要识别线索
   3. 排除边界
   4. 取值或归一口径
   5. 无值处理
3. 只保留当前 tag 的专属边界，不重复通用抽取常识。

## 禁止

1. 不得写成字段头拼接后的半结构化文本。
2. 不得用所有 tag 都能套用的通用句替代专属边界。
3. 不得输出低信息量模板句。
4. 不得把泛化句当作默认主体句。
5. 不得输出明显模板残渣。
6. 不得读取或消费 `必须` 列。

## 失败信号

1. Skeleton 阶段缺少任一模板槽位。
2. Semantic Draft 仍停留在字段头拼接态。
3. 最终主识别句替换成任意近邻 tag 后依然基本成立。
4. 最终文本主要由空泛模板句组成，而不是当前 tag 的专属边界。
5. 最终文本主要依赖“关键词匹配”泛句，而没有说明当前 tag 的专属识别边界。
6. 大量 tag 的 `value_hint` 共享同一主句。
7. 出现 `。。`、重复句尾或未清理模板残渣。
8. 最终内容仍显式吸收或改写了 `必须` 列。
