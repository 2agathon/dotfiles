## 一、操作权限边界

在动任何文件之前，先确认你有权限独立操作。

### 可以独立操作的文件

| 文件 | 允许的操作 |
|------|-----------|
| `parser/v1/namespaces/{ns}/page-types.data.json` | 新增、修改 category 和 type 条目；维护 `change_log` |
| `parser/v1/namespaces/{ns}/page-types.tree.json` | 与 `data.json` 同步更新（每次改 `data.json` 必须同步） |
| `parser/v1/namespaces/{ns}/types/{TYPE_ID}/page-semantic-spec.json` | 新增或修改标签和块类型定义 |
| `parser/v1/namespaces/{ns}/types/{TYPE_ID}/*.hint.md` | 新增或修改任意 hint 文件 |

### 不能修改的文件

| 文件 | 原因 |
|------|------|
| `meta/v1/page-types.schema.json` | 格式规范文件，由技术人员维护 |
| `meta/v1/page-semantic.schema.json` | 格式规范文件，由技术人员维护 |
| `meta/v1/templates/STRUCTURE_BLOCK_TEMPLATE_V1.md` | 模板文件，参考用，不需要修改（见第六章） |
| `default/types/TYPE_UNKNOWN/` | 系统保留兜底类型，不可修改 |
| `default/types/TYPE_OTHER/` | 系统保留兜底类型，不可修改 |

### 关于命名空间

本目录下存在多个命名空间（`default`、`other`、机构专属命名空间等）。**操作前应由技术人员告知你在哪个命名空间下工作**，不要跨命名空间修改。命名空间的设计原理见第四章。

## 二、常见任务流程

### 2.1 新增一个页类型

**第一步：在 `page-types.data.json` 添加类型条目**

找到合适的分类（`categories[]`），在其 `types[]` 数组末尾添加新类型。常用字段：`id`、`name`、`description`、`aliases`。详见第五章 5.1。

同时在顶层 `change_log` 数组**最前面**添加一条变更记录：
```json
{
  "date": "YYYY-MM-DD",
  "summary": "新增 TYPE_XXX（类型名）；一句话说明该类型的业务定义和锚点设计。"
}
```

**第二步：同步更新 `page-types.tree.json`**

`tree.json` 是 `data.json` 的轻量版本，供页类型识别步骤使用。**必须同步，否则 AI 识别时看不到新类型**。

`tree.json` 的每个 type 条目只需保留：`id`、`name`、`description`、`aliases`。

**第三步：创建类型目录和 `page-semantic-spec.json`**

在 `types/{TYPE_ID}/` 下新建目录，创建 `page-semantic-spec.json`。必须包含：`page_type`、`block_types`、`tags`。`action_paths` 可选，当前业务暂不需要定义。详见第五章 3.2。

**第四步：按需添加 hint 文件**

如果该类型在打标、组装、摘要上有特殊规则，按需创建对应的 `.hint.md` 文件。如果无特殊规则，可以不创建任何 hint 文件。详见第六章。

**第五步：验证**

用肉眼检查 JSON 格式是否正确（括号配对、逗号位置）。`*.schema.json` 是格式规范文件，当前不由工具强制校验，检查靠人工和编辑器。

---

### 2.2 优化某个类型的识别准确率

识别依赖 `page-types.data.json` 中该类型的 `description` 和 `aliases`。

- **识别混淆**（与另一个类型搞混）→ 完善 `description`，明确写出与该类型的区分点
- **别名覆盖不足**（某种叫法没识别到）→ 在 `aliases` 中补充该叫法
- **改完后同步 `tree.json`**（`description` 和 `aliases` 在两个文件里都存在）

---

### 2.3 改善某个步骤的 AI 处理质量

| 想改善的步骤 | 操作的文件 |
|-------------|-----------|
| 打标质量（标签取值不准） | `page-semantic-spec.json` 中对应标签的 `value_hint` |
| 打标质量（特殊排版/OCR 问题） | `tagging-hint.md` |
| 续写判定质量 | `continuation-hint.md` |
| 块组装质量（执行策略、字符控制、空块抑制） | `assembly-hint.md` |
| 块组装质量（块的语义边界、组成说明） | `block_types[].description` |
| 块摘要质量 | `block-summary-hint.md` |
| 页摘要质量 | `page-summary-hint.md` |

`page-semantic-spec.json` 的结构化字段和 `*.hint.md` 的关系：能用 JSON 字段表达的写 JSON，需要描述"处理策略、边界判断方法、常见错误预防"的写 hint 文件。

---

### 2.4 废弃一个类型

不要直接删除。在 `page-types.data.json` 中该类型条目里添加：
```json
"deprecated": true,
"replaced_by": "TYPE_新类型ID"
```
同步更新 `tree.json`，同步添加 `change_log` 记录。

## 三、核心概念

本章说明系统中的基本术语和整体链路。看不懂第二章的某个操作时，来这里查。

### 3.1 分类与类型（Category / Type）

页类型系统采用**两级树结构**：

- **分类（Category）**：一级，按业务领域划分。如"临床叙事记录"、"知情同意与告知"。分类本身不参与 AI 识别，只作为组织结构。
- **类型（Type）**：二级，是 AI 最终识别的粒度。如"入院记录"、"检验报告"。一个类型对应一类实际医疗文书。

**设计原则**：类型的粒度应当与"需要不同处理规则"的边界对齐，而不是与文书名称一一对应。如果两种文书在打标和块组装上的规则完全相同，它们应该合并为一个类型，用标签区分细节（例如"知情同意书类型"标签）。

### 3.2 命名空间（Namespace）

同一类型在不同机构、不同业务场景下，处理规格可能不同。**命名空间**是这个隔离层的名称。

- `default`：默认命名空间，覆盖所有标准医疗文书类型。
- 机构/场景专属命名空间：对 `default` 的局部 override——只覆盖需要差异化处理的类型，其余自动回退到 `default`。
- 独立领域命名空间（如 `other`）：与 `default` 平行，不继承 `default` 的类型，用于覆盖不同业务领域的文书。

**操作规则**：操作前由技术人员告知你在哪个命名空间工作。新建命名空间需要技术人员配合，不是日常操作。

### 3.3 标签、块类型、锚点（Tag / Block Type / Anchor）

**标签（Tag）**：从页面段落中抽取的最小语义单元。例如：`TAG_RECORD_TIME`（记录时间）、`TAG_CHIEF_COMPLAINT`（主诉）。

**块类型（Block Type）**：由多个标签实例组合而成的"材料块"，代表一个完整的医疗行为或事件。例如 `BLOCK_ADMISSION_INQUIRY`（入院记录-问诊）是由记录时间、主诉、现病史、既往史等标签组合而来的。一个块是业务上最小的可独立检索单元。

**锚点（Anchor）**：块的分组依据。当同一类型的文书在一份材料中出现多次时（如每次住院都有一份入院记录），需要按某个具有唯一性的标签把它们分开。这个标签就是锚点，通常是时间字段（如 `TAG_RECORD_TIME`）。

### 3.4 从原文到可检索材料的全链路

```
材料 PDF
    ↓ OCR + 分段
页面段落列表
    ↓ 步骤1：页类型识别
页面 → 类型 ID（如 TYPE_ADMISSION_RECORD）
    ↓ 步骤2：前页续写判定
续写段落标注
    ↓ 步骤3：段打标
标签实例列表（每段 → 0 或多个标签值）
    ↓ 步骤4：块组装（锚点分组 + AI 组合）
材料块列表（每块 = 一个医疗事件的完整信息）
    ↓ 步骤5：块摘要
每块 → 人类可读的连贯摘要
    ↓ 步骤6：页摘要
整页 → 事实型摘要（快速检索用）
```

## 四、文件组织

```
data/medical-material-specs/
│
├── meta/v1/
│   ├── 页类型创作指南.md                 ← 本文档
│   ├── page-types.schema.json            ← 页类型分类树的格式规范（不需要操作）
│   ├── page-semantic.schema.json         ← 语义规格文件的格式规范（不需要操作）
│   └── templates/
│       └── STRUCTURE_BLOCK_TEMPLATE_V1.md  ← 写 continuation-hint.md 的参考模板
│
└── parser/v1/namespaces/
    └── {namespace}/
        ├── page-types.data.json          ← 所有页类型的分类树（主数据）
        ├── page-types.tree.json          ← 轻量版分类树（页类型识别步骤使用，需与 data.json 同步）
        │
        └── types/
            └── {TYPE_ID}/
                ├── page-semantic-spec.json     ← 标签与块类型定义（结构化，必须）
                ├── continuation-hint.md   ← 续写判定提示（可选）
                ├── tagging-hint.md             ← 段打标提示（可选）
                ├── assembly-hint.md            ← 块组装提示（可选）
                ├── block-summary-hint.md       ← 块摘要提示（可选）
                └── page-summary-hint.md        ← 页摘要提示（可选）
```

**几个特殊文件说明：**

**`page-types.tree.json`**：与 `page-types.data.json` 共同维护，内容是后者的精简子集（只保留 `id`、`name`、`description`、`aliases`）。AI 在识别页类型时读这个文件，不读 `data.json`。两者不同步会导致识别结果与定义不一致，**每次修改 `data.json` 都必须同步更新 `tree.json`**。

**`STRUCTURE_BLOCK_TEMPLATE_V1.md`**：编写 `continuation-hint.md` 时的参考模板，定义了"结构块 + 版式重复段 + 跨页归属"的标准口径。写续写判定 hint 时建议参考。模板本身不需要修改（见第六章 6.1）。

**`*.schema.json`**：JSON Schema 格式规范文件，描述 `page-types.data.json` 和 `page-semantic-spec.json` 的合法字段结构。当前不由工具强制校验，不需要手动操作，也不应该修改。

**`TYPE_UNKNOWN/`、`TYPE_OTHER/`**（仅存在于 `default` 命名空间下）：系统保留的兜底类型，由流水线在特殊情况下自动使用。**不要修改这两个目录下的任何文件。**

## 五、字段参考

### 5.1 page-types.data.json

#### 顶层字段

| 字段 | 说明 |
|------|------|
| `version` | 文件版本号 |
| `description` | 本文件的一句话说明 |
| `last_updated` | 最后更新日期（格式 `YYYY-MM-DD`），每次修改后手动更新 |
| `schema_meta.owner` | 文件归属团队，不需要修改 |
| `schema_meta.maintainers` | 维护者列表，不需要修改 |
| `schema_meta.change_log` | 变更记录，**每次修改内容时在数组最前面添加一条**，格式见下文 |
| `sources` | 参考来源列表。如果新增类型有明确的国家规范或行业标准依据，在此登记；无依据时可保持不变或为空数组 |
| `categories` | 分类列表，包含所有页类型 |

**`change_log` 条目格式：**
```json
{
  "date": "YYYY-MM-DD",
  "summary": "新增 TYPE_XXX（类型名）；关键设计决策（锚点、跨页规则等）。"
}
```
`summary` 应该让未来的人看一眼就知道这次改了什么、为什么，不需要写完整句子，但要包含 Type ID。

#### 分类（category）字段

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | ✅ | 唯一 ID，格式 `CATEGORY_{英文大写}`，一旦使用不可更改 |
| `name` | ✅ | 中文展示名称 |
| `description` | ✅ | 说明这个分类覆盖哪类文书，以及与相邻分类的边界 |
| `order` | ✅ | 展示排序（整数），影响 AI 提示词中的出现顺序 |
| `source_keys` | 可选 | 参考来源的 key 列表，来源定义在顶层 `sources` 数组里 |
| `types` | ✅ | 该分类下的类型列表 |

#### 类型（type）字段

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | ✅ | 唯一 ID，格式 `TYPE_{英文大写}`，一旦有打标记录不可更改 |
| `name` | ✅ | 中文展示名称 |
| `description` | ✅ | **核心识别信号**。AI 依靠这个字段区分类型——见下方写法说明 |
| `aliases` | 推荐 | 常见别名列表，AI 基于页面标题匹配时使用。别名越全，兜底识别越好 |
| `source_keys` | 可选 | 参考来源的 key 列表 |
| `deprecated` | 可选 | `true` 表示已废弃，废弃时必须同时填写 `replaced_by` |
| `replaced_by` | 条件必填 | `deprecated: true` 时必填，指向替代类型的 ID |

**`description` 怎么写才有效：**

好的 `description` 包含：①这是什么文书（一句话定义）②典型内容或外观特征 ③与最容易混淆的类型的区分点。

```json
// 不好（太短、无边界）
"description": "住院记录"

// 好（有定义、有特征、有边界）
"description": "住院病例的首页摘要与编码入口，包含基本信息、诊断、手术/操作、费用等关键结构化信息。与入院记录的区别：病案首页是结构化编码表，入院记录是叙事性文本记录。"
```

### 5.2 page-semantic-spec.json

每个页类型一个文件，放在 `types/{TYPE_ID}/page-semantic-spec.json`。

#### 顶层字段

| 字段 | 说明 |
|------|------|
| `version` | 规格版本号 |
| `page_type.id` | 必须与所在目录名（`{TYPE_ID}`）一致 |
| `page_type.name` | 类型中文名，与 `page-types.data.json` 保持一致 |
| `action_paths` | 可选。块的情景/阶段/动作维度声明；未定义时块的维度字段存 null |
| `block_types` | 块类型定义数组 |
| `tags` | 标签定义数组 |

#### action_paths（可选）

声明这个类型下的块分别属于哪个"情景/阶段/动作"维度。**当前业务上暂不需要定义，可省略或留空数组。**

定义后的效果：
- 块组装时自动按 `block_type_id` 匹配并回填 context/phase/action 字段
- 人工编辑块时，所填的情景-阶段-动作会对照此列表做合法性校验

未定义时：块的 context/phase/action 字段存 null，人工编辑时不做白名单校验。

```json
{
  "context": { "id": "CTX_INPATIENT", "name": "住院" },
  "phase":   { "id": "PHASE_ADMISSION_CARE", "name": "入院收治" },
  "action":  { "id": "ACT_INQUIRY", "name": "问诊" },
  "block_type_ids": ["BLOCK_ADMISSION_INQUIRY"]
}
```

一个类型可以有多条 `action_paths`，每条对应一类不同性质的块。

#### block_types 字段

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | ✅ | 唯一 ID，格式 `BLOCK_{英文大写}` |
| `name` | ✅ | 中文名，用于摘要和界面显示 |
| `description` | ✅ | 块的语义说明：这种块的对象是什么、上下文结构关系是什么。**不写执行策略**，执行策略放 `assembly-hint.md` |
| `answerable_question` | 可选 | 本块设计用来直接回答的那一个问题（自然语言句子）。一个块只承诺回答一个问题，多问题请拆成多个块类型。 |
| `tags` | ✅ | 块包含的标签列表，每项有 `tag_id` 和可选的 `role` |

`tags[].role` 当前代码读取的值：`record_time_source`（时间锚点来源，用于块摘要中确定记录时间）。其余值（`document_root`、`node_anchor`、`node_meta`、`content`）已在规格中定义但代码暂未消费，填写不产生效果。

#### tags 字段

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | ✅ | 唯一 ID，格式 `TAG_{英文大写}` |
| `name` | ✅ | 中文名 |
| `value_hint` | ✅ | **AI 打标的核心依据**：应该抽取什么内容、从哪里取、取成什么格式、什么情况下不打 |
| `context_hint` | 可选 | AI 应在 `tag_context` 字段里补充什么说明（版面位置、判读依据等） |
| `anchor_binding` | 条件必填 | 锚点绑定配置，决定这个标签如何归属到一个块；仅对需要锚点归属的标签填写 |

**`value_hint` 写法模板：**
```
**可打本标签的内容：** [语义范围]。
**取值：** [抄取方式、规范化规则]。
**输出：** [单值/多值；格式；无值处理]。
**不打本标签：** [排除项]（如有）。
```

#### anchor_binding 子字段

| 子字段 | 说明 |
|--------|------|
| `target_tag_ids` | 锚点标签 ID，通常指向时间标签（如 `TAG_RECORD_TIME`） |
| `page_window.mode` | 锚点搜索范围。`within_type_run`：只在同类型连续段内搜索 |
| `direction` | 他项候选方向范围：`before` = 同段或前段，`after` = 同段或后段，`same_segment` = 仅同段，`nearby` = 同段及前后段。注意：这是候选范围约束，不等于最终命中的 `anchor_direction` |
| `required` | 锚点是否必须存在，`true` 时找不到锚点则不产出该标签的块归属 |
| `self_resolution` | 当标签本身就是锚点或允许自身参与候选时的选择策略：`self_only`（仅自身）、`self_first`（先自身再他项）、`direction_first`（先他项再自身）、`direction_only`（仅他项，默认值） |

## 六、Hint 文件写作规范

### 6.1 `continuation-hint.md`（续写判定）

**`meta/v1/templates/STRUCTURE_BLOCK_TEMPLATE_V1.md` 是写这个文件的参考模板。** 如果你要写 `continuation-hint.md`，先打开模板，把占位符替换为当前类型的字段线索词。模板覆盖了"结构块 + 版式重复段 + 跨页归属"的标准口径，大多数有结构块的文书类型可以直接套用。

如果一个类型没有这个文件，该类型的所有页会跳过续写判定步骤。

**应该写：**
- 结构单元的定义和典型特征
- 可跨页单元的判断规则
- 续写终止的触发条件
- 版式重复段的识别方法

**不应该写：** 标签取值规则（`value_hint` 的职责）、块的组装规则（`assembly-hint.md` 的职责）。

**篇幅建议：** 300–800 字。

---

### 6.2 `tagging-hint.md`（段打标）

补充打标时的处理策略，填补 `value_hint` 无法表达的部分。

**应该写：**
- 这种文档常见的 OCR/排版噪声及处理方式
- 多个标签边界容易混淆的情况及区分方法（如结构节点 vs 正文枚举编号）
- 特定字段的取值优先级规则
- 某类段落（目录页、版式重复段）的特殊处理方式
- 层级识别的易错点与防线

**不应该写：** 标签取值规则（`value_hint`）、如何组装块（`assembly-hint.md`）。

**篇幅建议：** 150–800 字（结构复杂、层级易混淆的类型可适当增加）。

---

### 6.3 `assembly-hint.md`（块组装）

块组装执行策略的主要文件。`block_types[].description` 只写块的语义说明，**所有执行策略写在这里**。

**应该写：**
- 空块抑制规则（什么情况下不生成块）
- 字符数统计口径和窗口拆分策略
- 归属过滤的执行顺序
- 正文与表格并存时的处理方式
- 容易产生冗余块或缺失块的特殊情况

**不应该写：** 块的语义定义（`block_types[].description`）、标签取值规则（`value_hint`）。

**篇幅建议：** 200–600 字。

---

### 6.4 `block-summary-hint.md`（块摘要）

**应该写：** 摘要的内容侧重点（哪些信息必须出现）、特殊格式要求、常见遗漏项。

**不应该写：** 通用摘要格式规则（在系统 Prompt 里）、块的组成规则（`block_types[].description`）。

**篇幅建议：** 100–300 字。

---

### 6.5 `page-summary-hint.md`（页摘要）

**应该写：** 哪些信息最应该出现在摘要里、推荐的摘要组织顺序、常见信息结构对应的摘要写法。

**不应该写：** 通用摘要格式规则、块的内容（摘要面向整页）。

**篇幅建议：** 100–300 字。

---

### 6.6 示范：TYPE_OFFICIAL_NORMATIVE_DOCUMENT

以下是规范性文书类型各 hint 文件的内容示范，供参考写法。

---

**`tagging-hint.md` 示范：**

```markdown
# 规范性文书 · 段打标处理指导

## 一、结构层级识别

1. **目录页**：若同一页内大量行呈现"编号 + 标题 + 页码"三元结构，该页为目录页，所有行均不打节点标签。
2. **标题与正文同行**：OCR 常将章节标题和其后第一句正文拼接在同一行（如"第一条 本办法适用于..."）。此时节点编号标签和内容单元标签分别独立打标，按分界截取，不得合并。
3. **括号式编号的层级判断**：形如"（一）""（二）"的括号式中文数字编号，无论数字大小，均为 L2 层级。

## 二、OCR 噪声处理

1. 章节编号中的中文大写数字（如"第三章""第五条"）应按 value_hint 规范化为"CHAPTER#3""ARTICLE#5"格式。
2. 页眉/页脚（文件编号、页码、单位抬头）通常不命中任何标签，按通用规则处理。
```

---

**`assembly-hint.md` 示范：**

```markdown
# 规范性文书 · 块组装处理指导

## 一、文本窗口块的拆分策略

当一个节点的正文超出单窗口上限（700 字）时，按原文顺序拆分为多个同类型块。拆分时以句号/分号为自然边界收口，不要在句子中间切断。续写到下一页的内容，如已通过续写判定确认归属本节点，纳入同一组装序列，不要因为跨页就另起一个块。

## 二、同一节点下正文与表格并存

如果一个节点下同时有正文和表格，应分别产出文本窗口块和表格块，两者并列，不是二选一。

## 三、归属限定的执行优先级

每个块类型的 description 中有明确的归属限定规则。这些规则是硬约束，比通用的"按 tag_id 收集"规则优先级更高。执行时先按 tag_id 收集候选，再用归属限定过滤。
```

---

**`block-summary-hint.md` 示范：**

```markdown
# 规范性文书 · 块摘要生成指导

1. **层级上下文必须出现**：摘要第一句应明确当前块所在的节点层级和编号（如"第三章第五条"）。
2. **条款内容优先于概括**：摘要应尽可能还原关键条款的字面含义，避免"规定了某某方面"这类空话。
3. **表格块的摘要**：说明表格的主题、行列维度和关键数值，不要逐行罗列。
```

---

**`page-summary-hint.md` 示范：**

```markdown
# 规范性文书 · 页摘要生成指导

1. **优先说明文档标识**：如果本页包含文档标题和施行日期，摘要第一句应包含这两项信息。
2. **按章节顺序组织**：用"第X章·X条"的简洁前缀标记各部分。
3. **对于续写页**：在开头注明"续前页"及所属章节。
```

## 七、LLM 消费原理

本章说明 AI 在每个步骤实际接收到什么数据，供技术人员和需要深入理解 AI 行为的人阅读。

### 流水线全景

| 步骤 | 步骤名 | 页类型已知？ | LLM 的主要任务 | 页类型定制提示词来源 |
|------|--------|------------|--------------|---------------------|
| 1 | 页类型识别 | ❌ 正在识别 | 判断这一页属于哪种类型 | `page-types.data.json` 中每个类型的 `description` 与 `aliases` |
| 2 | 前页续写判定 | ✅ 已知 | 判断本页哪些段落是上一页内容的延续 | `continuation-hint.md` |
| 3 | 段打标 | ✅ 已知 | 从段落中识别并抽取预定义的语义标签 | `tagging-hint.md` |
| 4 | 块组装 | ✅ 已知 | 把标签实例组合成材料块 | `assembly-hint.md` |
| 5 | 块摘要 | ✅ 已知 | 为每个材料块生成语义摘要 | `block-summary-hint.md` |
| 6 | 页摘要 | ✅ 已知 | 为整页生成事实型摘要 | `page-summary-hint.md` |

步骤 1 与其他步骤不同：识别时 AI 还不知道类型，需要看到所有类型的描述来做判断。步骤 2–6 发生在类型已知之后，AI 只处理当前类型的规格。

### 步骤 1：页类型识别

AI 收到：前一页内容（可选）、当前页全文、所有类型的描述清单（`name` + `description` + `aliases`）。任务：从清单里选出最匹配的类型。

| 数据 | 来源 | Prompt 位置 |
|------|------|------------|
| 当前页全文 | 数据库 `confirmed_page_content` | `{current_page_content}` |
| 前一页内容 | 数据库上一页记录 | `{prev_page_content}` |
| 类型描述清单 | `page-types.data.json` | `{type_descriptions}` |

### 步骤 2：前页续写判定

AI 收到：`continuation-hint.md` 的内容、前一页的所有段落、本页的所有段落。任务：判断本页每个段落是否是上一页内容的延续。

### 步骤 3：段打标

AI 收到：页类型信息、本页所有段落、标签集合（含 `value_hint`、`context_hint`）、`tagging-hint.md`（如有）。任务：逐段扫描，找出哪些段落命中哪些标签，从原文抽取对应的值。

### 步骤 4：块组装

AI 收到：锚点分组（服务层按 `anchor_binding` 预先计算）、未归属候选标签、`block_types` 定义、`assembly-hint.md`（如有）。任务：把分好组的标签实例按块类型定义组合成材料块。

### 步骤 5：块摘要

AI 收到：块的定位信息（类型名称、情景/阶段/动作、记录时间）、块内所有标签内容、`block-summary-hint.md`（如有）。任务：把标签实例转化成人类可读的连贯摘要。

### 步骤 6：页摘要

AI 收到：页类型信息、本页原文、本页所有块摘要（无块时传 `"无"`，始终传递）、`page-summary-hint.md`（如有）。任务：为整页生成事实型摘要，供快速检索和预览使用。

## 八、常见问题

**Q：新增类型时，`page-types.tree.json` 忘了更新会怎样？**

AI 识别页类型时只读 `tree.json`，不读 `data.json`。忘了更新意味着 AI 不知道这个类型存在，会把相关页面识别成其他类型，且不会报错。这是一个静默失效，不容易排查。**每次改 `data.json` 都要同步改 `tree.json`。**

**Q：hint 文件写错了会怎样？**

hint 文件是补充指导，不是硬约束。写错了不会导致程序报错，但可能导致 AI 输出质量下降。发现问题直接修改文件内容，下次调用即生效（系统有文件级缓存，文件修改后自动刷新）。

**Q：同一个类型在不同命名空间下的规格不同，怎么配置？**

在特定命名空间下创建该类型目录并放置文件即可。该命名空间没有某类型目录时，系统自动回退到 `default`。大多数情况下只需维护 `default`。

**Q：`change_log` 要不要维护？**

要。这是判断类型设计历史的唯一文本记录。每次修改类型定义（新增、调整字段、废弃）都在最前面加一条。不需要面面俱到，但 Type ID 和核心改动点必须出现。

**Q：类型 ID（`TYPE_*`）和标签 ID（`TAG_*`）能改名吗？**

**不能直接改名。** 一旦有页面或标签实例记录引用了这个 ID，改名会导致历史数据断裂。废弃类型应在 `page-types.data.json` 中标记 `deprecated: true` 并指定 `replaced_by`。标签 ID 的变更需要数据迁移，联系技术人员处理。

**Q：`sources` 数组要不要填？**

有明确的国家规范或行业标准依据时填（如原卫生部文件、行业标准号）。没有依据时保持为空数组即可，不影响运行。

**Q：`page-semantic-spec.json` 改了之后，缓存会立即更新吗？**

会。所有文件（spec.json 和 hint.md）都使用基于文件修改时间（mtime）的缓存策略：文件修改后，下次读取自动刷新，不需要重启服务。
