# chat-render

## 这个 skill 是什么

读取 chat-parser 的输出，AI 深度理解聊天内容，生成结构化分析结果，再由脚本渲染成 HTML 海报。

三件事，顺序不能乱：
1. **AI 分析**：读原始消息，理解这个群
2. **AI 输出分析 JSON**：作为参数传给渲染脚本
3. **脚本渲染**：render.py 接收参数，生成 HTML 文件

AI 不直接写 HTML，脚本不做任何内容判断。

**当前受众版本：群成员**——海报发回群里，触发「哈哈这说的就是我们」的感觉。后续待做：对方（私聊关系全貌）/ 自己（私人留存）/ 外人（让局外人理解这个群体）。

---

## 触发条件

- 用户说「生成海报」「出海报」「渲染」
- 用户已经有 parsed.json 文件
- chat-parser 刚刚执行完，用户想继续

---

## AI 行为规格

### Step 1. 获取数据

从 chat-parser 的 stdout 结果直接读取，或读取落盘的 parsed.json 文件。

确认数据包含 `messages` 字段（原始消息），否则无法做深度分析。

### Step 2. AI 深度分析

读完整个 `messages` 数组，生成「分析结果对象」。目标是质量，不是字数——够力就对了，废话不要。

```json
{
  "groupPersonality": "热闹型 | 专业型 | 温情型 | 混合型",
  // 必填 | 根据消息整体氛围判断，不能因为一条消息就定性

  "narrativeSummary": "一句话群画像",
  // 必填 | 让不认识这群人的人看完也能感觉到他们是什么样的 | 不能是「这是一个活跃的群」这种模板话 | 够短让人一眼记住，但不要为了短而削弱力度

  "mood": "本期整体情绪基调",
  // 必填 | 说这期整体的感觉，不是列举发生了什么

  "groupConsensus": "这群人心照不宣的共同观点",
  // 建议填 | 点到为止，不要解释 | 是群体共识不是某个人的立场

  "memberColors": {
    "成员名": "#c8402a"
    // 必填 | 给每位活跃成员分配一个颜色，贯穿整张海报
  },

  "memberAnalysis": {
    "成员名": {
      "role": "这个人在群里的角色",
      // 必填 | 具体，不是模板话 | 「每次都是第一个抛出问题的人」不是「话题发起者」
      "trait": "标志性行为特征",
      // 必填 | 有细节，能让本人看到会心一笑
      "quote": "原文摘取，不改写"
      // 必填 | 直接用原始 content 字段内容，不改写不意译
    }
  },

  "storyline": [
    {
      "title": "事件标题",
      // 必填 | 报纸头条感，让人想看下去
      "datetime": "2026-03-18T21:05:34+08:00",
      // 必填 | 事件发生时间
      "participants": ["成员名"],
      // 必填 | 涉及的成员
      "description": "叙述这件事",
      // 必填 | 说清楚起承转合就够，不要为了篇幅而堆砌 | 有脉络，不是孤立事件
      "keyMessage": "最能代表这件事的原始消息"
      // 必填 | 原文，不改写
    }
    // 数量：最少5条，最多12条 | 取重要节点，不要把小事和大事并列
    // 排序：默认时间顺序 | 如果非时间顺序叙事效果更好，可以调整，但要说明理由
  ],

  "storyVoices": [
    {
      "storyIndex": 1,
      // 对应 storyline 第几条，从1开始
      "speaker": "说话的人",
      // 不一定是故事主角，选最有话说但没说出口的那个
      "basis": "选这个人的推导依据",
      // 他在原始消息里的说话方式、他和这件事的关系
      "voice": "他没说出口的那句话"
      // 必须读原始消息找他的用词习惯和语气，用他自己的腔调写 | 不能用他不会说的词 | 要有点意思，没有意思就不写
    }
    // 整体可选——没有有意思的就不填，不要为了填而填 | 不是每条故事都要有
  ],

  "topicTurningPoints": [
    {
      "datetime": "2026-03-21T14:03:28+08:00",
      "sender": "成员名",
      "message": "那条改变对话方向的原始消息",
      "description": "这条消息怎么转变了话题"
      // 建议填 | 说清楚就够
    }
  ],

  "longestExchange": {
    "memberA": "成员名",
    "memberB": "成员名",
    "rounds": 6,
    "topic": "这次对话在聊什么",
    "startDatetime": "2026-03-18T21:05:34+08:00"
    // 建议填 | 两个人来回最多次的对话
  },

  "questionBoard": [
    {
      "question": "被问到的问题原文",
      "asker": "成员名",
      "responseCount": 3,
      "resolved": true
      // 建议填 | 引发讨论的问题，不是随口一问
    }
  ],

  "wordCloud": ["词1", "词2", "词3"],
  // 建议填 | AI 从消息里挑最能代表这期气质的词，不是机械词频 | 够有代表性就行，不要凑数量

  "funFact": "AI发现的有趣数据对比",
  // 建议填 | 具体，最好有数字，让人看完会「哦？」一下

  "unanswered": "那条没人回应的消息原文",
  "unansweredSender": "发送者昵称",
  // 建议填 | 沉了的消息，有点好笑或者有点意味深长的那种

  "offscreenVoice": "画外音"
  // 建议填 | AI 作为局外人说一句只有局外人才说得出的话
  // 风格：轻、有点好笑、回味有点什么 | 不分析具体人的关系和行为 | 不要正式，不要总结，就是一句感受
  // 参考：「看完了，没看懂，但感觉你们挺好的。」
}
```

### Step 3. 调用渲染脚本

把分析结果写成临时 JSON 文件，连同 parsed.json 路径一起传给脚本：

```bash
python <skill_path>/scripts/render.py \
  --parsed <parsed.json路径> \
  --analysis <analysis.json路径> \
  --output <输出目录>
```

### Step 4. 读取结果

脚本输出 HTML 文件路径到 stdout：

```
---CHAT_RENDER_RESULT---
<html文件路径>
---CHAT_RENDER_RESULT_END---
```

### Step 5. 告知用户

```
✓ 海报生成完成
  HTML：<路径>
  用浏览器打开，点「导出图片」保存 PNG
```

## 分析原则

- **只说消息里有的**：分析结果必须能在 messages 里找到依据，不能编造
- **金句原文摘取**：不改写，直接用原始 content 字段内容
- **故事线要有脉络**：体现事件之间的关联，不是孤立事件堆砌
- **角色描述要具体**：说出这个人具体怎么做的，不是贴标签
- **画像要有个性**：让不认识这群人的人看完也能感觉到他们是什么样的
- **内心独白原则**：读原始消息找说话方式，用他自己的腔调，不能用他不会说的词；没有有意思的就不写
- **画外音原则**：不分析具体人的关系和行为；轻、有点好笑；不正式不总结
- **奔着质量去**：字数不是目标，够力就对了

---

## 失效处理

| 情况 | 处理 |
|------|------|
| 没有 messages 字段 | 告知需要重新运行 chat-parser |
| 消息量 < 20 条 | 警告分析结果可能不准确，询问是否继续 |
| 消息全是媒体无文字 | 告知无法做深度分析，只输出统计版海报 |
| 脚本执行报错 | 输出错误信息，询问是否需要排查 |

## 输出规格

- 格式：HTML 文件（内嵌 CSS + JS，含「导出图片」按钮）
- 用户在浏览器打开 HTML，点按钮导出 PNG
- 宽度：520px
- 输出目录：parsed.json 所在目录下的 `chat-render/` 子目录
- 文件名：`{群名}_{开始日期}_{结束日期}_poster.html`

详细海报布局见 `references/poster-layout.md`。
