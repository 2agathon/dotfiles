description: >
  约束 AI 修改代码时的行为边界。确保改动范围可控、变更可见、不变量被保护。
  用户要求修改、修复、重构、调整现有代码时自动生效。
  不需要用户主动触发，AI 改代码时默认遵守底层约束，高风险改动时主动输出改动声明。
  不管新建文件（归 file-creation）、不管设计阶段（归 vibe-plan）。
---

## 生效条件

用户要求修改现有代码时自动生效。包括但不限于：
- "改一下"、"修一下"、"调整"、"重构"
- "这里有 bug"、"这个逻辑不对"
- "把 X 改成 Y"

不生效：
- 新建文件（归 file-creation）
- 设计阶段讨论（归 vibe-plan）
- 纯阅读或理解代码

---

## 底层约束（永远生效，不需要用户感知）

### 1. 手术式修改优先

默认只改用户指定的部分。不做以下任何事，除非用户明确要求：
- 重写整个文件
- 调整 import 顺序
- 改注释内容或格式
- 改代码风格或格式化
- "顺手优化"相邻代码

如果 AI 判断需要更大范围的改动才合理 → 停下来说明理由，等用户决定。

### 2. 范围膨胀必须报告

改 A 的过程中发现 B 也需要改 → 停，不默默改 B。输出：

```
[范围扩大] 改 A 需要联动修改 B，原因：{X}。
选项：1) 一起改  2) 只改 A，B 标记 TODO  3) 先停下来讨论
```

用户选择后才继续。

### 3. 不加不必要的防御代码

不主动添加用户没要求的：
- try-catch / error boundary
- null check / undefined guard
- fallback / default value
- 日志语句
- 类型守卫

如果 AI 认为某处确实需要防御 → 单独提出，说明理由，不默默加。

### 4. 变更可审

输出的代码必须让用户能快速看出改了什么：
- 小改动：只输出改动部分 + 足够定位的上下文
- 大改动：用注释标记改动位置（如 `// MODIFIED` 或 `// ADDED`）
- 不把三行改动淹没在一百行未改动代码中

### 5. 改后检查影响范围

改动完成后，AI 主动检查改动的影响范围：
- 找到改动部分的调用方、引用方、相关测试
- 确认没有被破坏
- 发现问题 → 报告具体问题和位置
- 没发现问题 → 简短说明已检查范围，如"已检查 X 处调用方，未发现破坏"

当代码库太大、AI 无法在当前上下文里完整检查时 → 退化为列出无法检查的区域，让用户决定是否需要进一步确认。

### 6. 改后跑测试

改动完成后，如果当前环境支持执行代码，且项目有现成测试：
- 自动跑相关测试套件
- 全绿 → 简短报告"已跑 X 个测试，全部通过"
- 有红 → 报告哪些测试红了，不自己修——测试红可能是测试该更新，也可能是代码真的坏了，这个判断给用户

当前环境不支持执行代码时 → 跳过，不报告此约束。不因为无法跑测试而阻塞交付。

## 改动声明（高风险时触发）

以下情况，AI 必须在动手前输出改动声明，等用户确认：

- 改动跨文件
- 改动涉及接口签名、公共方法、API 契约
- 改动会影响已知的调用方
- 用户的描述模糊，AI 需要自己推断改动范围

### 改动声明格式

```
改动范围：{哪些文件的哪些函数/区块}
不动范围：{同文件或相关文件中明确不碰的部分}
改动方式：手术式修改 / 局部重写 / 跨文件联动
预期副作用：{会影响哪些调用方} 或 无
```

### 不需要改动声明的情况

- 单文件内的明确改动
- 用户指定了具体改什么、怎么改
- 改动范围不存在歧义

这些情况直接改，底层约束仍然生效。

---

## Failure Modes

- 没有遵守手术式修改优先，重写了整个文件
- 默默改了用户没要求改的部分（静默行为变更）
- 范围扩大时没有停下来报告，默默联动修改了其他文件
- 主动加了用户没要求的防御代码
- 输出的代码无法快速看出改了什么
- 改动完成后没有检查影响范围
- 能检查的范围没检查，直接推给用户
- 环境支持执行且有测试，但改完后没跑
- 测试红了自己修了没告诉用户
- 高风险改动没有输出改动声明就开始改
- 低风险改动也弹改动声明，打扰用户
