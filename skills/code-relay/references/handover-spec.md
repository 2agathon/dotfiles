# 交接信内容规格

## 定义

交接信是 AI-to-AI 的上下文初始化通信。不是总结报告，不是介绍文档，不是面向人类的说明。

交接信的基本单位是代码区域。一个区域可以是一个模块、一个服务、一组相关文件——由用户的实际维护边界决定。

## 两种产物

### 区域交接信

面向单个代码区域，持续维护，跟着代码区域走。

### 工作上下文

面向跨区域任务，临时合并产物，任务结束后不保留。工作上下文除了包含各区域的关键信息外，还必须包含区域间的交叉点和依赖关系。

## 读者

零上下文的新会话中的 AI。

## 效果要求

新会话拿到交接信后应该：
1. 直接接管区域职责、核心对象链、高风险误读点
2. 只沿最短阅读路径做最小必要阅读
3. 快速形成自己的理解
4. 停在待命状态，等待指令

如果交接信会诱发新会话重新探索整体结构、重新发散寻找阅读入口、从零猜职责、或直接开始执行任务——交接信不合格。

## 必须包含的内容

- 区域一句话定义（跨区域时每个区域各一句）
- 核心对象链
- 主流程 / 主依赖链
- 高风险误读点
- 已确认事实
- 待复核判断

跨区域工作上下文额外必须包含：
- 区域间依赖关系
- 本次任务涉及的交叉点

## 按情况包含的内容

有就写，没有不硬凑：

- 真相分层（事实底稿 / 规则本体 / 派生结果）
- 区域边界（处理什么、不处理什么）
- 最近演化主线
- 最短继续阅读路径

## 格式要求

- 放在单独代码块中
- 可直接复制给零上下文会话
- 不掺杂当前会话的讨论噪音
- 不使用人类汇报口吻
- 不复述思考过程
- 不写元话语（"给下一个 AI"、"这份文本的作用是"）
- 语气是直接面向另一个智能程序员的工作通信

## 标注体系

继续使用 identity skill 的标注：
- `[ASSUMPTION: 内容]` — 推断的内容
- `[OPEN QUESTION: 问题]` — 未解决的问题
- `[PREMISE: 内容]` — 依赖但未被检验的前提
