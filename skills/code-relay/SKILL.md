---
name: code-relay
description: >
  代码的会话间上下文接力。管三件事：认识会话怎么输出交接信、执行会话怎么加载交接信、执行会话怎么收工回流认识。
  支持单区域交接信的持续维护，也支持跨区域多份交接信的合并。
  交接信保存到项目 `.ai/relay/` 下，执行会话可直接从文件加载，无需手动复制。
  用户说"认识会话"、"扫描模块"、"生成交接信"时触发认识模式。
  用户贴了交接信并说"开始工作"、"接手"、"继续"时触发加载模式。
  用户说"加载 {区域名}"时从文件加载交接信。
  用户说"收工"、"结束"、"打包"时触发收工模式。
  只用于代码的会话接力，不用于普通对话。
---

## 触发条件

三种模式，互斥：

| 信号 | 模式 |
|------|------|
| 用户说"认识会话"/"扫描模块"/"生成交接信" | 认识 |
| 用户贴了交接信 + 说"开始"/"接手"/"继续" | 加载 |
| 用户说"加载 {区域名}"/"继续 {区域名}"（未贴交接信） | 加载（从文件读） |
| 用户说"收工"/"结束"/"打包" | 收工 |

不触发：
- 普通对话、认识探索、非代码讨论
- 用户只是在讨论代码但没有要求会话接力

---

## 模式一：认识

### 输入

必须有：
- 代码材料（目录、文件、文档、commit 记录，由用户提供）

可选：
- 一份或多份旧交接信（用户贴入，或从 `.ai/relay/` 读取已有文件）
- 上一轮执行摘要（用户贴入，或从 plan 文件的执行日志中读取）

如果 `.ai/relay/` 目录存在，AI 应主动检查是否有该区域的已有交接信，作为增量更新的骨架。

### AI 行为

#### Step 0：判断输入形态

交接信的基本单位是**代码区域**——可以是一个模块、一个服务、一组相关文件，由用户的实际维护边界决定。区域交接信持续维护，不随任务消亡。跨区域任务时，合并多份交接信产出临时工作上下文，任务结束后不保留。

先判断当前输入属于哪种情况，再走对应路径：

| 输入形态 | 路径 |
|----------|------|
| 纯材料，无旧交接信 | 从零建模，输出该区域的交接信 |
| 材料 + 一份旧交接信 | 增量更新，以旧交接信为骨架，用新材料校验和更新 |
| 材料 + 多份旧交接信 | 跨区域合并，先理解各区域，再产出一份面向当前任务的工作上下文，同时按需更新各区域交接信 |
| 材料 + 执行摘要（有无旧交接信均可） | 吸收执行摘要中的发现和临时判断，体现在新交接信中 |

#### Step 1：识别认知入口

不预设阅读顺序。先判断当前材料里哪些文件最能最快回答：

1. 这些代码负责什么
2. 依赖什么真相源
3. 核心对象链是什么
4. 主流程或关键依赖链是什么
5. 最近在往哪里演化

输出"最短阅读路径"及选择理由。

增量更新时：先加载旧交接信作为认知骨架，再用新材料校验和更新，不从零建模。
有执行摘要时：必须吸收其中的"发现了什么跟预期不一样的"和"临时判断"。

#### Step 2：形成认知骨架

基于选择的入口建立初始认知。先有结构，再填细节。

跨区域时：先分别理解各区域，再识别区域之间的依赖关系和交叉点。

#### Step 3：校验

- 文档和代码是否一致
- 设计目标和实现现状是否漂移
- 最近 commit 在收什么口径、补什么机制

#### Step 4：输出理解快照

以下为必须项：

- **一句话定义**——这个区域（或这组区域）负责什么
- **边界**——涉及哪些区域，各自负责什么，区域之间的依赖关系
- **核心对象链**
- **主流程 / 主依赖链**
- **高风险误读点**——至少 3 条
- **事实 / 判断分栏**——已确认事实 vs 待复核判断

以下为按材料情况选写——有就写，没有不硬凑：

- 真相分层（事实底稿 / 规则本体 / 派生结果）
- 最近演化主线
- 最短继续阅读路径

跨区域时额外必须项：
- **区域间交叉点**——这次任务会碰到哪些区域之间的接缝
- **各区域交接信的更新建议**——哪些旧交接信需要更新、更新什么

#### Step 5：生成交接信并保存到文件

根据 Step 0 判断的输入形态输出：

| 输入形态 | 输出 |
|----------|------|
| 单区域（新建或更新） | 该区域的交接信 |
| 跨区域 | 一份面向当前任务的工作上下文 + 各区域交接信的更新建议（用户决定是否回去更新） |

交接信的完整规格见 `references/handover-spec.md`。

**保存到文件：**

单区域交接信生成后，保存到 `.ai/relay/{区域名}.md`，目录不存在则创建。文件格式：

```markdown
# Relay · {区域名}
更新时间：{时间}

---

{交接信正文}
```

已有同名文件时直接覆盖——旧版本靠 git 历史追溯。

跨区域的临时工作上下文不保存到文件——它跟着任务走，任务结束后不保留。如果用户要求更新各区域交接信，按单区域逻辑分别保存。

### 禁止事项

在完成理解快照之前：
- 不提修改方案、优化建议、重构建议
- 不进入实现取舍辩论
- 不把推断写成既定事实
- 不替未来执行会话执行任务

## 模式二：加载

### 输入

- 用户贴的一份或多份交接信
- 或用户贴的一份工作上下文（跨区域合并产物）
- 或用户指定区域名，AI 从 `.ai/relay/{区域名}.md` 读取

### AI 行为

#### Step 1：加载

如果用户贴了交接信 → 读取贴入的内容。
如果用户说了区域名但没贴 → 从 `.ai/relay/` 下查找对应文件。找不到 → 告知用户没有该区域的交接信。

读取交接信内容，接管其中的区域职责、核心对象链、高风险误读点。

多份交接信时：分别接管，识别区域间的依赖关系。

#### Step 2：最小校验

如果交接信指定了"最短阅读路径"，沿该路径做最小必要阅读，验证关键事实是否仍然成立。

发现不一致 → 列出不一致项，标注 `[PREMISE: 交接信说 X，代码现状是 Y]`，等用户判断。

#### Step 3：加载确认

输出极短的确认。单区域不超过 10 行，多区域每个区域不超过 3 行：

单区域时：

```
区域职责：{一句话}
核心对象：{列出}
高风险点：{列出}
不一致项：{有则列，无则省略}
待命中。
```

多区域时：

```
区域 A 职责：{一句话}
区域 B 职责：{一句话}
核心对象：{列出}
区域间依赖：{一句话}
高风险点：{列出}
不一致项：{有则列，无则省略}
待命中。
```

#### Step 4：停

等待用户指令。不主动建议、不发散、不开始执行。

### 禁止事项

- 不重新探索区域整体结构
- 不重新发散寻找阅读入口
- 不从零猜区域职责
- 不主动开始执行任务

## 模式三：收工

### 触发时机

用户说"收工"/"结束"/"打包"，或会话中执行任务已完成时。

### AI 行为

输出执行摘要，严格三段，放在代码块中。

单区域时：

```
## 做了什么
改了哪些文件，加了什么，删了什么。事实陈述，不解释为什么。

## 过程中发现了什么跟预期不一样的
交接信说的和实际代码不一致的地方、文档和实现的漂移、意外的依赖关系。
没有就写"无"。

## 临时判断
这次为了推进做了哪些临时决定，哪些需要后续确认。
没有就写"无"。
```

跨区域任务时，第一段按区域分组：

```
## 做了什么
### 区域 A
改了哪些文件...
### 区域 B
改了哪些文件...

## 过程中发现了什么跟预期不一样的
...

## 临时判断
...
```

执行摘要的完整规格见 `references/execution-summary-spec.md`。

### 关键约束

- 读者是下一轮认识会话，不是人
- 格式是可直接贴给新认识会话的代码块
- 不写成人类汇报稿
- 不复述思考过程
- 不写元话语

### 保存到文件

如果当前执行有对应的 plan 文件（`.ai/plans/` 下）→ 将收工摘要追加到 plan 文件的"执行日志"段。
如果没有对应的 plan 文件 → 只在对话中输出，不单独保存。

---

## Failure Modes

- 认识模式：为了凑格式硬编选写项内容
- 认识模式：有执行摘要输入但没有吸收其中的发现
- 认识模式：在理解快照完成前就提修改建议
- 认识模式：跨区域输入时没有识别区域间依赖，把所有材料当一个区域处理
- 认识模式：把不内聚的代码硬塞进"一个区域"的框架
- 认识模式：生成交接信后没有保存到文件
- 认识模式：`.ai/relay/` 下已有该区域交接信但没有读取，从零建模
- 加载模式：没等用户指令就开始执行
- 加载模式：重新从零探索而不是接管交接信
- 加载模式：用户指定了区域名，文件存在但没有去读，反而问用户贴交接信
- 收工模式：执行摘要写成了人类汇报稿
- 收工模式：遗漏了过程中发现的不一致或临时判断
- 收工模式：跨区域任务时没有按区域分组，混在一起写
- 收工模式：有对应 plan 文件但没有追加执行日志
