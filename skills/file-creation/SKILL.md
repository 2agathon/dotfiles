## 触发条件

- 用户要新建文件或目录
- 用户在犹豫新内容放哪里
- 用户主动说"audit"

不触发：
- 用户只是在讨论方案，还没有要建任何文件
- 用户在阅读或理解现有代码

---

## 创建时

### Step 1：确定相关目录范围

从新文件的职责关键词出发，在当前上下文里找：
- 名字和职责关键词有直接重叠的目录
- 新文件的直接父目录
- 如果是顶层目录，读根目录一层

**关键读判断标准：** 职责关键词来自用户描述。用户说"配置加载"，就找名字里有 config、setting、env 的目录。找不到相关目录 → 跳过边界自检，直接进 Step 2。

没有现有结构 → 从当前语言惯例推断放置位置，推断到目录层级为止，不推断具体结构，标注 `[INFERRED from {language} convention]`，推断有歧义时列出两个候选，不自己选。

### Step 2：命名自检

用三个信号判断名字是否有足够边界：

1. 这个名字里有没有边界模糊的角色词？角色词的特征是：什么都能往里塞，说不出它不做什么。结合当前语言判断。有 → 追问：修饰词够不够窄，能不能排除掉不相关的职责？
2. 能否用一句话说完"它只做 X，不做 Y"？说不完整 → 名字有问题
3. 和同层已有文件比，命名粒度是否一致？粒度差距大 → 名字有问题

名字有问题 → 停，告诉用户具体哪个信号触发了，不自己改名。

### Step 3：边界自检

对照 Step 1 读到的目录：
- 有没有已有文件的职责和新文件高度重叠？

有重叠 → 列出重叠文件和重叠的职责部分，问用户新建还是并入。
没有 → 直接创建，不打扰。

## 用户犹豫放哪里时

这是唯一能从对话里识别的熵信号——用户说"这个放哪好"、"这里还是那里"。

从犹豫描述里提取职责关键词，读相关目录，列出候选：

```
候选A：{路径} — 理由：{和新文件职责的重叠点}
候选B：{路径} — 理由：{同上}
```

不推荐，只列出，等用户决定。

---

## Audit 时

### Step 1
问用户要看哪个范围。没有指定不展开。

### Step 2
读指定范围，找两种症状：

```
[症状：名字和职责不符]
文件：{路径}
名字说的是：{X}
里面实际有：{Y}

[症状：边界渗漏]
职责：{Z}
散落在：{路径1} / {路径2} / {路径3}
```

没有症状 → 直接说"指定范围内未发现症状"。

---

## Guardrails

- 推断语言惯例时不推断到具体文件结构
- 边界自检只读相关目录，不全量扫描
- 用户犹豫时只列候选，不推荐

---

## Failure Mode

- 给出了"标准结构"或"建议结构"
- 自己改了名字而没有告诉用户
- 移动了文件而没有等用户确认
- 全量扫描目录而不是关键读
- 两个自检都没问题但还是打扰了用户
- Audit 时没有问范围就展开
