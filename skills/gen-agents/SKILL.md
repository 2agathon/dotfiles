---
name: gen-agents
description: Use when starting work on any project, when no AGENTS.md exists, when the user needs to set up project-level AI context, or when joining a project with existing rules. Guides identification of where Layer 0 default behavior needs adjustment for this project, and where that adjustment should live based on ownership.
---

## 这个 skill 做什么

找出 Layer 0 在这个项目里哪里不够用，把缺口表达成行为指令。

判断标准：AGENTS.md 里每一句话，都应该能回答——
**在什么情况下，AI 应该做什么不同于 Layer 0 默认行为的事？**

回答不了这个问题的，不属于这里。

---

## 执行逻辑
```
第一步：确认所有权（必须问，不可推断）
    有所有权 → 走 Layer 2 路径
    没有所有权 → 走 Layer 3 路径

Layer 2 路径：
    第二步：收集语言约定（必须问，不可推断）
    第三步：识别 Layer 0 例外（引导用户，不是让用户自己想）
    第四步：识别项目独有约束（引导用户，不是收集项目描述）
    第五步：确认技术栈（用于校准措辞，不写进输出）
    第六步：生成，自检，交付

Layer 3 路径：
    第二步：确认是否有团队规范（必须问，不可推断）
    第三步：生成对应形态，交付
```

---

## 必须问的信息

以下信息不可推断，不可假设，必须等用户回答后再继续。

**所有路径都需要：**

> "在这个项目里，你能决定工程规范并让其他人遵守吗？"

**Layer 2 路径需要：**

> "语言约定是什么？（代码注释 / 日志输出 / 表注释 / 错误码描述 / Git commit message 分别用中文还是英文）"

**Layer 3 路径需要：**

> "这个项目有没有现成的团队规范或 AGENTS.md？"

---

## Layer 2 路径：有所有权

**引导识别缺口，不是收集项目信息**

语言约定问完之后，按顺序引导以下三个缺口识别。
能从对话或文件里提取的直接提取，不重复问。

**缺口一：Layer 0 的哪些规则在这个项目里不成立？**

不能让用户凭空想，主动给引导：

> "Layer 0 有这些默认规则，逐条说明在这个项目里是否成立：
> - 不用 dict / Map 作为返回类型
> - 业务逻辑和 IO 分离
> - 敏感字段进日志前必须脱敏
> - 权限校验统一在入口层
> 有没有哪条在这个项目里需要调整？"

没有例外就不写这节。

**缺口二：这个项目有什么 Layer 0 没覆盖的硬约束？**

引导用户识别行为约束，不是项目描述：

> "这个项目有没有 Layer 0 没说过的特殊规则？
> 比如：某类操作必须经过特定检查、某类数据有合规要求、某个接口有安全边界。
> 描述这类'在什么情况下必须做什么'的约束。"

区分标准对用户说清楚：
- ✓ "任何 SQL 执行前必须通过 guardrail 检查"——行为约束
- ✗ "这个项目用了 DuckDB"——项目描述，不属于这里

**缺口三：技术栈是什么？**

不写进输出，用来校准措辞：
"不用 Map 作为返回类型"在 Python 里写成"不用 dict"，在 Java 里写成"不用 HashMap"。

从对话或文件推断，推断不出来再问。

**输出结构**

只有内容才有节，没有内容不生成空节。
`[OPEN QUESTION]` 和 `[ASSUMPTION]` 就近挂在对应条目下，不单独成节。
```
# AGENTS.md · [项目名]

## 语言约定
代码注释：
日志输出：
表注释 / 字段注释：
错误码描述：
Git commit message：

## Layer 0 在这里的例外
[每条说明原因]
// DEVIATION: 原因

## 这个项目独有的行为约束
[每条是行为指令：在什么情况下，做什么，为什么]
```

**自检**

- [ ] 每一条能否回答"在什么情况下，AI 做什么不同于 Layer 0 的事"？
- [ ] 有没有把项目描述（技术栈、架构、端口、目录结构）写进来？

## Layer 3 路径：没有所有权

**情况 A：有团队规范，打补丁**
```
# AGENTS.local.md · [项目名] · 个人补丁

## 覆盖团队规范的地方
[每条说明原因]
// DEVIATION: 原因

## 团队规范没说但需要的行为约束
[每条是行为指令：在什么情况下，做什么，为什么]
```

**情况 B：没有规范，也没权建**
```
# AGENTS.local.md · [项目名] · 个人工作层

[临时文件，等正式规范建立后迁移废弃]

## 在这个项目里的工作方式
[不是项目规范，是 Layer 0 在这个项目里的个人具体化]
[同样只写行为指令，不写项目描述]
```

**两种情况都适用**

临时妥协表不在初始生成物里。项目进行中遇到"违反 principles 但改不动"时，手动追加：
```
## 临时妥协表
| 位置 | 违反了什么 | 原因 | 计划 |
|------|-----------|------|------|
```

**Layer 3 的硬约束**

不进仓库。放项目根目录，加进 `.gitignore`。
没有 `.gitignore` 时用 `.git/info/exclude`。
进了仓库就变成 Layer 2，性质完全不同。

---

## 生成完成后

**Layer 2：**

> "已生成 AGENTS.md，复制到项目根目录并提交。
> 文件里只有行为指令，没有项目描述——这是正确的。
> 随项目推进发现新的 Layer 0 缺口，直接追加到对应节。"

**Layer 3：**

> "已生成 AGENTS.local.md，放项目根目录，确认已加进 `.gitignore`。
> 遇到违反 principles 但改不动的地方，追加到临时妥协表。
> 等正式规范建立后，把有价值的条目迁移进去，这个文件可以废弃。"
