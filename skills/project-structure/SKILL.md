## 这个 skill 做什么

在生成任何文件路径或目录之前，先对齐结构和命名约定。
目录结构是业务地图，不是技术分层的刻板复制。

判断标准：陌生人打开项目，30秒内能说出这个项目的业务地图。

## 触发判断

以下情况触发：
- 生成任何新文件之前
- 用户说"新建一个模块"、"加一个服务"、"创建这个功能的目录"
- 发现现有文件放错了层或域

## 信息收集

**需要什么信息**
- 业务域是什么、文件属于哪一层、文件的职责是什么

**信息从哪里来**
- 用户提供 → 用户描述要做什么，直接从描述里提取
- 读代码 → 提供现有目录结构，推断业务域划分和层次约定
- 从对话提取 → 已经讨论过业务域和模块划分，从对话里提取

**信息缺口怎么处理**
- 信息已经在对话里 → 直接提取，不重复问
- 业务域不明确 → 必须问，域决定顶层目录
- 层次不明确 → 按标准结构推断（api/domain/infra），用 `[ASSUMPTION]` 标注
- 在已有项目里 → 先读懂现有结构意图，再决定放哪里
- 完全新项目 → 先生成顶层目录骨架，确认后再生成文件

## 标准结构
```
src/
├── {domain}/               # 业务域，小写单数
│   ├── api/                # 对外接口层：Controller、DTO、Request、Response
│   ├── domain/             # 领域层：Service、Entity、Validator、规则
│   ├── infra/              # 基础设施层：Mapper、外部SDK适配、Repository实现
│   ├── docs/               # 这个模块的文档，跟着模块走
│   │   ├── decisions/      # 决策记录：YYYY-MM-DD-[标题].md
│   │   ├── api.md          # 接口文档
│   │   └── architecture.md # 架构说明（复杂模块才需要）
│   └── {Domain}Module.java # 模块入口，声明这个域对外暴露什么
└── shared/                 # 跨域共享，谨慎放东西
    ├── exception/
    ├── config/
    └── util/
```

## 规则

### 目录反映业务域，不是技术角色

顶层目录用业务名词：`claim`、`policy`、`underwriting`。
不用 `service`、`controller`、`utils` 作为顶层目录。

**这条规则在保护什么**
保护业务地图的可读性。技术分层作为顶层目录，打开项目只能看到"有 service 层、有 controller 层"，看不出这个系统做什么业务。业务域作为顶层，打开就知道系统覆盖哪些域。

**不遵守会发生什么**
`service/` 目录里塞了 40 个类，没有人能说清楚哪些属于核心业务、哪些是辅助逻辑。新功能不知道加在哪里，最后全塞进 `service/`，目录彻底失去导航价值。

**什么情况下可以偏离**
接手遗留项目，存量代码已经按技术分层组织，短期内不做整体重构。新文件仍按业务域放，加注释说明意图。

**偏离时记录**
```
// DEVIATION: 遗留项目保留技术分层结构，新模块 claim/ 按业务域组织
```

**自检**
- [ ] 顶层目录名是业务名词还是技术角色名（service / controller / utils）？
- [ ] 陌生人看这个目录结构，能否 30 秒内说出系统做什么业务？

---

### 文档跟着模块走

每个业务域的文档放在 `{domain}/docs/` 里，不集中放根目录 `docs/`。

**这条规则在保护什么**
保护文档和代码的生命周期一致性。集中放的文档最终都会和代码脱节——模块改了，没有人记得去根目录 `docs/` 更新对应文件。

**不遵守会发生什么**
根目录 `docs/` 里的 `claim-api.md` 描述的是六个月前的接口，现在的接口已经完全不同了，但没有人知道文档过期了。新人按文档接入，调试两小时才发现文档是错的。

**什么情况下可以偏离**
跨域的架构决策、部署文档、整体系统说明放根目录 `docs/`，这些本来就不属于单个模块。

**偏离时记录**
不适用——跨域文档放根目录是正确行为，不是偏离。

**自检**
- [ ] 这份文档描述的是某个业务域内的内容，还是跨域的系统级内容？
- [ ] 文档放的位置和它描述的代码是否在同一个目录树下？

### 命名用业务名词，单数

目录：业务名词，小写，单数。`claim` 不是 `claims`，`policy` 不是 `policies`。
文件：业务名词 + 技术角色。`ClaimValidator`、`PolicyRepository`、`UnderwritingService`。
禁止：`Manager`、`Handler`、`Helper`、`Processor`——这些词没有边界，什么都能塞。

**这条规则在保护什么**
保护命名的边界声明能力。`ClaimValidator` 只能做校验，`ClaimManager` 可以做任何事。名字越模糊，边界越容易腐化。

**不遵守会发生什么**
`PolicyHandler` 出现半年后，里面有校验、有转换、有通知、有日志。没有人能说清楚它的职责，也没有人敢动它。

**什么情况下可以偏离**
无合理偏离场景。如果找不到合适的业务名词，说明这个模块的职责还没想清楚，先想清楚再建目录。

**自检**
- [ ] 这个名字能否用一句话说清楚它只做什么，不做什么？
- [ ] 名字里有没有 Manager / Handler / Helper / Processor？

---

### 新增文件按规范放，不按现有风格放

已有项目里现有文件全在 `service/` 不代表新文件也要放那里。新文件按域放，加注释说明。

**这条规则在保护什么**
防止错误风格自我复制。存量代码的风格会污染新生成的文件——不主动抵抗，就默认继承。

**不遵守会发生什么**
遗留项目的混乱结构在每次新增文件时被强化一次，永远没有改善的机会。

**什么情况下可以偏离**
团队明确决定暂不调整结构，统一按旧风格维护一致性，等待专项重构。这必须是显式决策，不是默认行为。

**偏离时记录**
```
// DEVIATION: 团队决定暂维持旧结构一致性，待 Q3 重构时统一迁移
```

**自检**
- [ ] 这个文件的放置位置是根据业务域判断的，还是因为其他文件也放在这里？

---

### 修改文件就地改，不顺手重构

改一个方法不等于重构整个文件。没有明确授权，不动修改范围之外的任何代码。

**这条规则在保护什么**
保护变更的可预期性。"顺手"改了范围外的代码，reviewer 不知道哪些是必要变更、哪些是顺手改动，review 成本翻倍，出了问题也不知道是哪行改的。

**不遵守会发生什么**
修一个 bug 的 PR 里顺带重命名了十个变量、调整了三个方法顺序。出了新问题不知道是 bug fix 引入的还是重构引入的。

**什么情况下可以偏离**
用户明确说"顺便重构一下"，或者发现放错的文件影响当前修改的正确性。

**偏离时记录**
```
// DEVIATION: 修复范围外的命名调整，原名 processData 含义不清，已在此处统一
```

**自检**
- [ ] 这次改动有没有触碰用户没有提到的代码？
- [ ] 如果有，是否得到了明确授权？

---

### 发现放错的文件：标注，不立刻移动

在注释或 `[OPEN QUESTION]` 里记录，等合适时机统一处理。

**这条规则在保护什么**
防止隐性的高成本变更。移动文件会破坏 import 链，可能影响多个模块，需要统一处理而不是随手移。

**不遵守会发生什么**
顺手移动了一个文件，三个其他模块的 import 断掉，但 AI 不知道，继续生成，最后编译失败，排查成本远高于标注等待。

**什么情况下可以偏离**
文件刚创建（当次会话内），还没有任何依赖，可以立刻移到正确位置。

**自检**
- [ ] 这个文件是否已经被其他模块引用？
- [ ] 如果需要移动，是否已经用 `[OPEN QUESTION]` 标注？

## 生成文件前的三个问题

在生成任何文件之前必须能回答：
1. 这个文件属于哪个业务域？
2. 它在这个域里是哪一层（api / domain / infra / docs）？
3. 它的职责是什么，一句话说清楚？

三个问题答不上来，说明边界还没想清楚，不要生成。
