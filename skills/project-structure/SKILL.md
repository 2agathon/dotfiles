## 这个 skill 做什么

在生成任何文件路径或目录之前，对齐结构和命名约定。
目录结构是业务地图，不是技术分层的刻板复制。

---

## 强制停止点

以下情况 AI 必须停下来，等用户确认后才能继续——不能 ASSUMPTION 绕过：

- **生成文件前的三个问题**：答案必须来自用户，不接受 AI 自行推断后继续
- **顶层目录命名**：业务域名称必须来自用户，不能从技术栈推断
- **发现疑似放错位置的文件**：列出具体文件和理由，等用户确认是否确实放错，不能自行判定
- **命名违反规则**：发现 Manager / Handler / Helper / Processor，停下来列出问题等用户决定命名，不能自己选一个继续

---

## 生成文件前的三个问题

**这三个问题的答案必须来自用户，不接受 AI 自行推断：**

1. 这个文件属于哪个业务域？
2. 它在这个域里是哪一层（api / domain / infra / docs）？
3. 它的职责是什么，一句话说清楚？

> "在生成之前，我需要确认三件事：[列出三个问题]。"

三个问题答不上来，说明边界还没想清楚，不要生成。

**信息来源的优先级：**
1. 用户直接说明 → 直接用
2. 用户已在对话里描述过 → 提取后向用户确认："我理解是 X，对吗？"
3. 无法从对话得出 → 必须问，不能推断

**可以 ASSUMPTION 的（实现细节）：**
- 层次划分（api / domain / infra），用 `[ASSUMPTION]` 标注，用户可随时纠正

**不能 ASSUMPTION 的（必须来自用户）：**
- 业务域名称
- 顶层目录结构
- 文件是否放错位置的判定

## 标准结构
```
src/
├── {domain}/               # 业务域，小写单数
│   ├── api/                # 对外接口层：Controller、DTO、Request、Response
│   ├── domain/             # 领域层：Service、Entity、Validator、规则
│   ├── infra/              # 基础设施层：Mapper、外部SDK适配、Repository实现
│   ├── docs/               # 这个模块的文档，跟着模块走
│   │   ├── decisions/      # 决策记录：YYYY-MM-DD-[标题].md
│   │   ├── api.md
│   │   └── architecture.md
│   └── {Domain}Module.java
└── shared/
    ├── exception/
    ├── config/
    └── util/
```

## 规则

### 目录反映业务域，不是技术角色

顶层目录用业务名词：`claim`、`policy`、`underwriting`。
不用 `service`、`controller`、`utils` 作为顶层目录。

**这条规则在保护什么**
保护业务地图的可读性。技术分层作为顶层，打开项目只能看到有哪些层，看不出这个系统做什么业务。

**不遵守会发生什么**
`service/` 目录里塞了 40 个类，没有人能说清楚哪些属于核心业务、哪些是辅助逻辑。

**什么情况下可以偏离**
接手遗留项目，存量代码已按技术分层组织，短期不做整体重构。新文件仍按业务域放。

**偏离时记录**
```
// DEVIATION: 遗留项目保留技术分层结构，新模块 claim/ 按业务域组织
```

**自检（AI 执行，有疑问时停下来问用户）**
- [ ] 顶层目录名是业务名词还是技术角色名？
- [ ] 陌生人看这个结构，能否 30 秒内说出系统做什么业务？说不出 → 停下来，等用户重新确认域划分。

---

### 文档跟着模块走

每个业务域的文档放在 `{domain}/docs/`，不集中放根目录 `docs/`。

**这条规则在保护什么**
保护文档和代码的生命周期一致性。集中放的文档最终都会和代码脱节。

**不遵守会发生什么**
根目录 `docs/` 里的接口文档描述的是六个月前的接口，没有人知道它过期了。

**什么情况下可以偏离**
跨域的架构决策、部署文档、整体系统说明放根目录 `docs/`——这不是偏离，是正确行为。

**自检（AI 执行）**
- [ ] 这份文档描述的是某个域内的内容，还是跨域的系统级内容？

---

### 命名用业务名词，单数

目录：业务名词，小写，单数。`claim` 不是 `claims`。
文件：业务名词 + 技术角色。`ClaimValidator`、`PolicyRepository`。
禁止：`Manager`、`Handler`、`Helper`、`Processor`——这些词没有边界，什么都能塞。

**这条规则在保护什么**
保护命名的边界声明能力。名字越模糊，边界越容易腐化。

**不遵守会发生什么**
`PolicyHandler` 出现半年后，里面有校验、有转换、有通知、有日志。没有人能说清楚它的职责。

**什么情况下可以偏离**
无合理偏离场景。找不到合适的业务名词，说明职责没想清楚，先想清楚再建目录。

**自检（AI 执行，发现违反时停下来，等用户决定命名）**
- [ ] 名字里有没有 Manager / Handler / Helper / Processor？有 → 停下来，不能自己选名字继续。
- [ ] 这个名字能否用一句话说清楚它只做什么、不做什么？

---

### 新增文件按规范放，不按现有风格放

已有项目里现有文件全在 `service/` 不代表新文件也要放那里。
新文件按域放，加注释说明意图。

**这条规则在保护什么**
防止错误风格自我复制。存量代码的风格会污染新生成的文件。

**偏离时记录**
```
// DEVIATION: 团队决定暂维持旧结构一致性，待 Q3 重构时统一迁移
```

---

### 修改文件就地改，不顺手重构

没有明确授权，不动修改范围之外的任何代码。

**这条规则在保护什么**
保护变更的可预期性。顺手改了范围外的代码，reviewer 不知道哪些是必要变更。

**不遵守会发生什么**
修一个 bug 的 PR 里顺带重命名了十个变量。出了新问题不知道是 bug fix 引入的还是重构引入的。

**什么情况下可以偏离**
用户明确说"顺便重构一下"。或发现放错的文件影响当前修改的正确性——但必须先列出来等用户确认，不能自行决定移动。

**偏离时记录**
```
// DEVIATION: 修复范围外的命名调整，原名 processData 含义不清，已在此处统一
```

**自检（AI 执行）**
- [ ] 这次改动有没有触碰用户没有提到的代码？有 → 列出来，等用户授权再动。

---

### 发现疑似放错的文件：列出，等确认，不自行判定
```
[OPEN QUESTION: {文件路径} 疑似放错位置，理由是 {原因}。是否需要处理？]
```

**这条规则在保护什么**
防止 AI 基于自己的工程判断静默影响用户对代码库状态的认知。

**什么情况下可以偏离**
文件刚创建（当次会话内），还没有任何依赖，可以立刻移到正确位置——但仍需告知用户。

**自检（AI 执行）**
- [ ] 这个文件是否已经被其他模块引用？
- [ ] 如果需要移动，是否已用 `[OPEN QUESTION]` 标注并等待用户确认？
