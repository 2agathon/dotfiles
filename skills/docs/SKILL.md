# Docs

## Purpose

`docs` 用于编写和补充**说明性文档**。

它处理的不是"所有和文档有关的东西"，而是那些主要服务于以下目标的内容：

- 让读者知道这是什么
- 让读者知道怎么用
- 让读者知道怎么改
- 让读者知道当前状态是什么
- 让读者在最短路径内完成理解、使用、维护

它不负责完整保存决策历史。
当内容的核心变成"为什么当时选了这个而不是别的"，应移交给 `decision-record`。

---

## Core Principle

文档的价值不在"写得多完整"，而在于：

**让特定读者，以最短路径，完成特定任务。**

写文档前先问三件事：

- 这份文档是给谁看的
- 读者读完后要能做什么
- 这份文档不打算覆盖什么

如果这三件事不清楚，先不要写长文。先把文档边界钉住。

---

## 强制停止点

以下情况必须停下来，等用户确认，不能推断后继续：

- **读者不是从用户的明确表述中得出的**：必须问，不能从对话推断后跳过确认
- **新建任何文档**：必须走三件事确认流程，不能跳过
- **"局部补充"的认定有疑问**：必须问用户，不能自己决定是否属于局部补充

### 局部补充的边界

**局部补充的定义：已有文档中，修改范围不超过一个小节，且读者和目的不变。**

以下情况不属于局部补充，必须走确认流程：
- 新建任何文档
- 修改范围跨越多个小节
- 读者或目的发生变化

**局部补充的认定由用户决定，不由 AI 自行判断。有疑问时必须问。**

### 开始写之前确认三件事

仅当新建文档时走这个流程，局部补充豁免。

> "在开始之前，我需要确认三件事：读者是谁、目的是什么、不覆盖什么。"

1. **读者是谁**：工程师 / 产品 / 客户 / 运维 / 未来的自己
2. **目的是什么**：读者需要用这份文档做什么决定或完成什么任务
3. **不覆盖什么**：这份文档明确不处理什么，防止文档膨胀

**读者和目的的来源优先级：**
1. 用户明确说明 → 直接用
2. 用户在对话里描述过 → 提取后确认："我理解读者是 X，目的是 Y，对吗？"
3. 无法从对话得出 → 必须问，不能推断

## 信息收集

**必须来自用户，不能 ASSUMPTION：**
- 读者是谁（用户没有明确说明时）
- 目的是什么（用户没有明确说明时）

**可以从对话或代码推断，用 `[ASSUMPTION]` 标注：**
- 文档结构
- 覆盖深度

**信息缺口处理：**

| 情况                       | 处理方式                                   |
| -------------------------- | ------------------------------------------ |
| 信息已在对话里             | 直接提取，确认后继续                       |
| 局部补充（认定由用户决定） | 直接做，不走确认流程                       |
| 读者和目的已明确           | 只补其他缺口                               |
| 读者不明确                 | 必须问，不能推断                           |
| 目的不明确                 | 必须问，不能推断                           |
| 原材料不足                 | `[OPEN QUESTION]` 标注缺失部分，其余先生成 |
| 完全从零开始               | 走三件事确认流程                           |

---

## Scope

本 skill 处理以下内容：

- 当前对象是什么
- 当前对象如何组织
- 当前对象如何使用
- 当前对象如何维护
- 当前对象如何修改
- 当前对象目前有哪些限制或注意事项
- 读者进入对象时，最应该先看到什么

本 skill 不处理以下内容：

- 完整决策历史
- 备选方案比较史
- 当时取舍的完整展开
- 决策为何成立、何时失效的独立记录
- 会议纪要、讨论转录
- 认识漂移与张力演化本身

这些内容应视情况移交给：

- `decision-record`
- `notes-protocol`

## Decide the Writing Mode

先区分：这次是在**补现有文档**，还是**新建文档**。

### 补现有文档

适用于：

- 某个文档已经存在
- 当前只是补一段说明、修正某个过时部分、补一个缺口

此时目标不是重写文档，而是：

- 找到最合适的插入点
- 保持原有结构稳定
- 只补当前缺的那部分
- 避免把局部补充写成全文重构

### 新建文档

适用于：

- 当前对象没有合适文档承载
- 现有文档结构无法容纳目标内容
- 当前要服务的读者与现有文档完全不同

此时先定：文档标题、读者、目标、覆盖范围、推荐阅读顺序。

---

## Reader-Oriented Writing

按不同读者，调整写法。

| 读者        | 语气             | 深度 | 结构重点                   |
| ----------- | ---------------- | ---- | -------------------------- |
| 工程师      | 直接、技术       | 深   | 为什么这样设计、边界在哪里 |
| 产品 / 业务 | 清晰、无术语     | 浅   | 做了什么、影响是什么       |
| 客户        | 友好、结果导向   | 极浅 | 能做什么、怎么用           |
| 运维        | 步骤清晰、可操作 | 中   | 怎么做、出了问题怎么办     |
| 未来的自己  | 诚实、完整       | 深   | 当时为什么这样决定         |

## Common Document Types

### README

README 是地图，不是百科全书。

它至少应帮助读者快速回答：

- 这个项目是什么
- 适合谁用
- 如何启动
- 核心目录在哪里
- 进一步该看什么

README 不应承担：完整模块文档、完整接口手册、完整决策历史、全部实现细节。

**自检（AI 执行）**
- [ ] 陌生人读完第一段，能否说出这个项目是做什么的？说不出 → 重写第一段。
- [ ] README 里有没有可以放到其他文档里的细节？有 → 移出去，保持 README 简洁。

---

### 技术文档

适用：接口文档、架构说明、模块设计。

- 说清楚为什么这样设计，不只是是什么
- 边界在哪里：这个模块 / 接口不处理什么
- 有 `[ASSUMPTION]` 的地方显式标出
- 涉及历史分叉时引用对应 DR，不在此展开

**自检（AI 执行）**
- [ ] 文档里有没有说"为什么这样设计"，还是只描述了"是什么"？
- [ ] 这个模块 / 接口"不处理什么"有没有写清楚？

---

### 需求文档

适用：功能说明、业务流程。

- 先说业务目标，再说实现细节
- 每个需求说清楚：谁、在什么情况下、需要做什么、验收标准是什么
- `[OPEN QUESTION]` 标出还没确认的需求
- 如果需求过程中产生了真实决策分叉，应把分叉落到 DR，不要在需求文档里无限堆叠历史取舍

**自检（AI 执行）**
- [ ] 每条需求有没有明确的验收标准，还是只有描述？没有 → 停下来，问用户。
- [ ] 还没确认的需求有没有用 `[OPEN QUESTION]` 标注？

---

### 维护文档

适用：运维手册、排查指南。

重点写：常见问题、排查顺序、关键日志、高风险改动点、恢复方式、常见误区。

它服务的是"恢复与维护"，不是"保存完整历史"。

**自检（AI 执行）**
- [ ] 出了问题，运维能否只看这份文档完成排查？不能 → 补缺口。
- [ ] 有没有把历史取舍塞进来？有 → 移交给 DR，这里只保留引用。

---

### 上手文档 / Onboarding

重点写：推荐阅读顺序、第一周最该知道什么、哪些部分先别深挖、哪些地方容易迷路、哪些文档是下一跳入口。

它的目标是"把人带进来"，不是"一次讲完全部真理"。

**自检（AI 执行）**
- [ ] 新人读完，知道第一天该做什么吗？不知道 → 加一节"从这里开始"。
- [ ] 有没有一上来就塞了太多内部历史？有 → 移到后面或单独文档。

---

## Boundary with `decision-record`

### 什么时候内容不再属于 `docs`

先问一句：

**如果删掉"为什么"这一段，读者还能不能理解和使用系统？**

- 如果**还能**，这部分通常不该留在 `docs`，而应进入 `decision-record`
- 如果**不能**，这部分仍属于说明文档的一部分

再问一句：

**这段内容未来会不会因为约束变化而失效，但当前现状本身暂时不变？**

- 如果**会**，它更适合写成 `decision-record`

### 在 `docs` 中如何引用 `decision-record`

说明文档不扛完整决策史，只保留简洁引用：

- 当前采用同步调用。决策背景见 `DR-003`
- 当前不抽公共组件。相关取舍见 `DR-011`
- 当前保留兼容层。历史原因见 `DR-007`

### 禁止越界展开

当用户要的是说明文档时，不要把完整备选方案、完整取舍过程、完整决策脉络塞进正文。除非这些内容直接影响当前理解或操作，否则只做一句引用。

---

## Boundary with `notes-protocol`

如果内容主要是判断如何慢慢变化、理解如何重组、张力如何移动，那么这不属于 `docs`。应进入 `notes-protocol`。

在 `docs` 中只保留当前读者完成任务所必需的结论，不展开完整认识演化过程。

---

## Writing Rules

### 先写地图，再写百科

优先给读者方向感。让读者先知道从哪里进入、什么最重要、哪些部分暂时不用看。不要一上来堆满细节。

### 同一件事只在一个地方说清楚

如果某件事已经有合适文档承载，当前文档只做引用，不重复完整展开，不制造多个版本的说明。

### 优先写结构性信息

职责、边界、入口、依赖、风险点、修改路径——比起大段实现细节，这些更能帮助读者真正进入对象。

### 不要把文档写成代码注释扩写版

文档应该补代码看不出来的东西：对象职责、为什么先看这里、哪些边界不能轻易跨、哪些限制当前成立。

### 不要把文档写成完整历史回忆录

说明文档只保留对当前理解有帮助的历史。内容一旦变成"为什么当时选了这个而不是别的"，移交给 `decision-record`。

---

## Execution Rules

1. **先判断文档是否真的需要存在**：不是所有内容都值得新建文档。如果一句补充能解决，就不要新建大文。
2. **先判断应该补哪里，而不是先写新文**：如果现有文档已经有合适位置，优先补进去。
3. **不要默认"越多越好"**：只写当前读者完成当前任务真正需要的内容。
4. **先写对象，再写细节**：先给结构，再给展开。不要一开始就掉进细节泥潭。
5. **出现分叉时立刻停下来判断边界**：一旦内容开始转向备选项比较、当时约束、为什么不是别的，判断是否应移交给 `decision-record`。
6. **不要让说明文档承担完整历史负担**：文档应帮助进入对象，而不是成为历史档案库。

---

## Failure Modes

- 把 README 写成百科全书
- 把模块文档写成代码注释复述
- 把说明文档写成讨论纪要
- 把历史取舍整段塞进当前文档
- 同一件事在多个地方重复展开
- 明明是局部补充，却重写整篇
- 明明已经越界到决策历史，却不移交给 DR
- 明明是认识演化，却硬塞进说明文档

出现以上情况时，应收缩范围，重新确认读者、目的和边界。

---

## Success Criteria

一个合格的说明文档，应满足以下标准：

- 目标读者能快速知道自己该看什么
- 读者读完后能完成预期任务
- 文档边界清楚，不假装覆盖一切
- 当前对象的职责、入口、限制说得清楚
- 不重复、不膨胀、不替代别的文档
- 遇到历史分叉时，能正确引用 DR
- 遇到认识演化时，不假装自己在处理 notes-protocol 的事情

---

## Default Behavior

如果用户没有额外要求，默认：

- 先明确读者、目标和不覆盖什么
- 优先补现有文档，而不是无脑新建
- 保持简洁，只写当前对象必须写清的内容
- 遇到决策历史时引用 `decision-record`
- 遇到认识漂移时不硬塞进文档
- 让文档成为入口，而不是负担

---

## Final Reminder

`docs` 的工作，不是替系统保存一切。它的工作是：

**把当前对象写清楚，让正确的读者，顺利进入、使用、维护它。**

当内容开始变成"为什么当时选了这个而不是别的"，不要硬扛。把它移交给 `decision-record`，然后在这里正确引用。
