## 这个 skill 做什么

根据读者和目的，生成对应的文档。
文档的价值不在于完整，在于让读者用最短路径完成他需要做的事。

## 触发判断

以下情况触发：
- 用户说"帮我写文档"、"写个说明"、"记录一下这个决定"
- 用户完成了一个功能或讨论，需要沉淀成文字
- 项目缺少 README、接口文档、决策记录
- 用户指向过去某个讨论或决定，需要补记录

## 信息收集

**需要什么信息**
读者是谁、目的是什么、不覆盖什么、原材料在哪里。

**信息从哪里来**

- 用户提供 → 用户把需求、对话记录、代码、决策背景直接提供
- 读代码 → 提供代码或目录结构，推断出技术文档、README、接口说明
- 从对话提取 → 讨论完一个功能或决策，直接从这段对话里提取，不要让用户重新描述

**信息缺口怎么处理**

| 情况                               | 处理方式                                      |
| ---------------------------------- | --------------------------------------------- |
| 信息已在对话里                     | 直接提取，不重复问                            |
| 局部补充（加一个字段、改一段说明） | 直接做，不走确认流程                          |
| 读者和目的已知                     | 只补缺口，不重新确认                          |
| 读者不明确                         | 必须问——读者决定语气、深度、结构              |
| 目的不明确                         | 必须问——目的决定文档要让读者做什么            |
| 原材料不足                         | 用 `[OPEN QUESTION]` 标注缺失部分，其余先生成 |
| 完全从零开始                       | 走下面的三件事确认流程                        |

**"局部补充"的边界：** 局部补充是"已有文档的小范围修改"。如果是新建一份文档、或者方向不明确，不属于局部补充，必须确认读者和目的。不能用"局部补充"来跳过所有确认。

## 开始写之前确认三件事

仅当完全从零开始新文档时走这个流程：

1. **读者是谁** → 工程师 / 产品 / 客户 / 运维 / 未来的自己
2. **目的是什么** → 他需要用这份文档做什么决定或完成什么任务
3. **不覆盖什么** → 这份文档明确不处理什么，防止文档膨胀

**这三件事在保护什么**
保护文档的有效性。没有明确读者和目的的文档最终变成"看起来完整但没人用"的东西——写的时候面面俱到，读的时候找不到想要的信息。

**不确认就开始会发生什么**
写了一份给工程师看的接口文档，结果读者是产品经理，里面全是技术细节，产品经理看不懂，还得重写一份。

## 按读者调整写法

| 读者        | 语气             | 深度 | 结构重点                   |
| ----------- | ---------------- | ---- | -------------------------- |
| 工程师      | 直接、技术       | 深   | 为什么这样设计、边界在哪里 |
| 产品 / 业务 | 清晰、无术语     | 浅   | 做了什么、影响是什么       |
| 客户        | 友好、结果导向   | 极浅 | 能做什么、怎么用           |
| 运维        | 步骤清晰、可操作 | 中   | 怎么做、出了问题怎么办     |
| 未来的自己  | 诚实、完整       | 深   | 当时为什么这样决定         |

## 决策记录

决策记录的目的：几个月后回来，能在最短时间还原当时的判断逻辑。
```
## [决策标题]

日期：
状态：已决定 / 已废弃 / 待确认

### 背景
当时面对什么问题，为什么需要做这个决定。

### 考虑过的选项
1. [选项A] - 优点 / 缺点
2. [选项B] - 优点 / 缺点

### 最终决定
选了什么，为什么。

### 代价和风险
这个决定放弃了什么，带来了什么风险。

### [OPEN QUESTION]
当时没有解决的问题，留给未来。
```

**这个结构在保护什么**
保护决策的可追溯性。没有决策记录，三个月后没有人知道为什么这样设计，也没有人敢改——改了怕踩坑，不改技术债持续积累。

**自检**
- [ ] "背景"里有没有说清楚当时面对的约束，还是只说了"我们决定做 X"？
- [ ] "代价和风险"有没有诚实地写，还是空着？

## 技术文档

适用：接口文档、架构说明、模块设计。

- 说清楚为什么这样设计，不只是是什么
- 边界在哪里：这个模块 / 接口不处理什么
- 有 `[ASSUMPTION]` 的地方显式标出
- 和代码保持同步，代码变了文档必须跟着变

**自检**
- [ ] 文档里有没有说"为什么这样设计"，还是只描述了"是什么"？
- [ ] 这个模块 / 接口"不处理什么"有没有写清楚？

---

## 需求文档

适用：功能说明、业务流程。

- 先说业务目标，再说实现细节
- 每个需求说清楚：谁、在什么情况下、需要做什么、验收标准是什么
- `[OPEN QUESTION]` 标出还没确认的需求

**自检**
- [ ] 每条需求有没有明确的验收标准，还是只有描述？
- [ ] 还没确认的需求有没有用 `[OPEN QUESTION]` 标注，还是当成已确认写进去了？

---

## README

适用：项目、模块、仓库的入口文档。

- 30秒内让陌生人知道：这是什么、怎么用、去哪里找更多信息
- README 是地图，不是百科全书

**自检**
- [ ] 陌生人读完第一段，能否说出这个项目是做什么的？
- [ ] README 里有没有可以放到其他文档里的细节，让 README 本身保持简洁？

## 文档腐化防止

文档最大的问题不是写不好，是写完之后和现实脱节。

- 决策记录：状态字段必须维护，废弃的决定标"已废弃"
- 技术文档：接口变了、架构变了，文档当天更新
- README：每次项目结构大变动后过一遍

---

## 注意

- 不在读者和目的确认之前开始写（局部补充除外）
- 不因为"看起来完整"就堆砌内容
- 同一件事只在一个地方说清楚
- 不让用户重新描述已经在对话里的信息
- "局部补充"豁免条件只适用于已有文档的小范围修改，不适用于新建文档

## 输出位置

文档放在对应业务模块的 `docs/` 目录下，路径约定见 `project-structure` skill。
