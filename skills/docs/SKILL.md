## 这个 skill 做什么

根据读者和目的，生成对应的文档。
文档的价值不在于完整，在于让读者用最短路径完成他需要做的事。

## 触发判断

以下情况触发：
- 用户说"帮我写文档"、"写个说明"、"记录一下这个决定"
- 用户完成了一个功能或讨论，需要沉淀成文字
- 项目缺少 README、接口文档、决策记录
- 用户指向过去某个讨论或决定，需要补记录

## 信息收集

**我需要什么信息**
- 读者是谁、目的是什么、不覆盖什么、原材料在哪里

**信息从哪里来**
- 你提供 → 你把需求、对话记录、代码、决策背景直接给我
- 读代码 → 提供代码或目录结构，推断出技术文档、README、接口说明
- 从对话提取 → 你们讨论完一个功能或决策，直接从这段对话里提取信息

**信息缺口怎么处理**
- 信息已经在对话里 → 直接提取，不重复问
- 用户指向过去的内容 → 从对话回溯提取，不要让用户重新描述
- 只是局部补充（加一个字段、改一段说明）→ 直接做，不走完整确认流程
- 读者和目的已知 → 只补缺口，不重新确认
- 读者不明确 → 必须问，读者决定语气、深度、结构
- 目的不明确 → 必须问，目的决定文档要让读者做什么
- 原材料不足 → 用 [OPEN QUESTION] 标注缺失部分，其余先生成，缺口留占位符
- 完全从零开始 → 才走完整的三件事确认流程

## 开始写之前确认三件事

仅当完全从零开始时走这个流程：

1. **读者是谁** → 工程师 / 产品 / 客户 / 运维 / 未来的自己
2. **目的是什么** → 他需要用这份文档做什么决定或完成什么任务
3. **不覆盖什么** → 这份文档明确不处理什么，防止文档膨胀

## 按读者调整写法

| 读者       | 语气             | 深度 | 结构重点                   |
| ---------- | ---------------- | ---- | -------------------------- |
| 工程师     | 直接、技术       | 深   | 为什么这样设计、边界在哪里 |
| 产品/业务  | 清晰、无术语     | 浅   | 做了什么、影响是什么       |
| 客户       | 友好、结果导向   | 极浅 | 能做什么、怎么用           |
| 运维       | 步骤清晰、可操作 | 中   | 怎么做、出了问题怎么办     |
| 未来的自己 | 诚实、完整       | 深   | 当时为什么这样决定         |

## 决策记录（最重要）

决策记录的目的：几个月后回来，能在最短时间还原当时的判断逻辑。
```
## [决策标题]

日期：
状态：已决定 / 已废弃 / 待确认

### 背景
当时面对什么问题，为什么需要做这个决定。

### 考虑过的选项
1. [选项A] - 优点 / 缺点
2. [选项B] - 优点 / 缺点

### 最终决定
选了什么，为什么。

### 代价和风险
这个决定放弃了什么，带来了什么风险。

### [OPEN QUESTION]
当时没有解决的问题，留给未来。
```

## 技术文档

适用：接口文档、架构说明、模块设计。

- 说清楚为什么这样设计，不只是是什么
- 边界在哪里：这个模块/接口不处理什么
- 有 [ASSUMPTION] 的地方显式标出
- 和代码保持同步，代码变了文档必须跟着变

## 需求文档

适用：功能说明、业务流程。

- 先说业务目标，再说实现细节
- 每个需求说清楚：谁、在什么情况下、需要做什么、验收标准是什么
- [OPEN QUESTION] 标出还没确认的需求

## README

适用：项目、模块、仓库的入口文档。

- 30秒内让陌生人知道：这是什么、怎么用、去哪里找更多信息
- README 是地图，不是百科全书

## 文档腐化防止

文档最大的问题不是写不好，是写完之后和现实脱节。

- 决策记录：状态字段必须维护，废弃的决定标"已废弃"
- 技术文档：接口变了、架构变了，文档当天更新
- README：每次项目结构大变动后过一遍

## 注意

- 不要在读者和目的确认之前开始写（除非局部补充）
- 不要因为"看起来完整"就堆砌内容
- 同一件事只在一个地方说清楚
- 不要让用户重新描述已经在对话里的信息

## 输出位置

文档放在对应业务模块的 `docs/` 目录下，路径约定见 `project-structure` skill。
