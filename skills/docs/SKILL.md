## 这个 skill 做什么

根据读者和目的，生成对应的文档。
文档的价值不在于完整，在于让读者用最短路径完成他需要做的事。

---

## 强制停止点

以下情况必须停下来，等用户确认，不能推断后继续：

- **读者不是从用户的明确表述中得出的**：必须问，不能从对话推断后跳过确认
- **新建任何文档**：必须走三件事确认流程，不能跳过
- **"局部补充"的认定有疑问**：必须问用户，不能自己决定是否属于局部补充

---

## 局部补充的边界

**局部补充的定义：已有文档中，修改范围不超过一个小节，且读者和目的不变。**

以下情况不属于局部补充，必须走确认流程：
- 新建任何文档
- 修改范围跨越多个小节
- 读者或目的发生变化

**局部补充的认定由用户决定，不由 AI 自行判断。有疑问时必须问。**

---

## 开始写之前确认三件事

仅当新建文档时走这个流程，局部补充豁免。

> "在开始之前，我需要确认三件事：读者是谁、目的是什么、不覆盖什么。"

1. **读者是谁**：工程师 / 产品 / 客户 / 运维 / 未来的自己
2. **目的是什么**：读者需要用这份文档做什么决定或完成什么任务
3. **不覆盖什么**：这份文档明确不处理什么，防止文档膨胀

**读者和目的的来源优先级：**
1. 用户明确说明 → 直接用
2. 用户在对话里描述过 → 提取后确认："我理解读者是 X，目的是 Y，对吗？"
3. 无法从对话得出 → 必须问，不能推断

**这三件事在保护什么**
保护文档的有效性。没有明确读者和目的的文档最终变成"看起来完整但没人用"——写的时候面面俱到，读的时候找不到想要的信息。

---

## 信息收集

**必须来自用户，不能 ASSUMPTION：**
- 读者是谁（用户没有明确说明时）
- 目的是什么（用户没有明确说明时）

**可以从对话或代码推断，用 `[ASSUMPTION]` 标注：**
- 文档结构
- 覆盖深度

**信息缺口处理：**

| 情况                                 | 处理方式                                   |
| ------------------------------------ | ------------------------------------------ |
| 信息已在对话里                       | 直接提取，确认后继续                       |
| 局部补充（定义见上，认定由用户决定） | 直接做，不走确认流程                       |
| 读者和目的已明确                     | 只补其他缺口                               |
| 读者不明确                           | 必须问，不能推断                           |
| 目的不明确                           | 必须问，不能推断                           |
| 原材料不足                           | `[OPEN QUESTION]` 标注缺失部分，其余先生成 |
| 完全从零开始                         | 走三件事确认流程                           |

---

## 按读者调整写法

| 读者        | 语气             | 深度 | 结构重点                   |
| ----------- | ---------------- | ---- | -------------------------- |
| 工程师      | 直接、技术       | 深   | 为什么这样设计、边界在哪里 |
| 产品 / 业务 | 清晰、无术语     | 浅   | 做了什么、影响是什么       |
| 客户        | 友好、结果导向   | 极浅 | 能做什么、怎么用           |
| 运维        | 步骤清晰、可操作 | 中   | 怎么做、出了问题怎么办     |
| 未来的自己  | 诚实、完整       | 深   | 当时为什么这样决定         |

## 决策记录

决策记录的目的：几个月后回来，能在最短时间还原当时的判断逻辑。
```
## [决策标题]

日期：
状态：已决定 / 已废弃 / 待确认

### 背景
当时面对什么问题，为什么需要做这个决定。

### 考虑过的选项
1. [选项A] - 优点 / 缺点
2. [选项B] - 优点 / 缺点

### 最终决定
选了什么，为什么。

### 代价和风险
这个决定放弃了什么，带来了什么风险。

### [OPEN QUESTION]
当时没有解决的问题，留给未来。
```

**自检（AI 执行）**
- [ ] "背景"里有没有说清楚当时面对的约束，还是只说了"我们决定做 X"？
- [ ] "代价和风险"有没有诚实地写，还是空着？空着 → 停下来，问用户。

## 技术文档

适用：接口文档、架构说明、模块设计。

- 说清楚为什么这样设计，不只是是什么
- 边界在哪里：这个模块 / 接口不处理什么
- 有 `[ASSUMPTION]` 的地方显式标出
- 和代码保持同步，代码变了文档必须跟着变

**自检（AI 执行）**
- [ ] 文档里有没有说"为什么这样设计"，还是只描述了"是什么"？
- [ ] 这个模块 / 接口"不处理什么"有没有写清楚？

---

## 需求文档

适用：功能说明、业务流程。

- 先说业务目标，再说实现细节
- 每个需求说清楚：谁、在什么情况下、需要做什么、验收标准是什么
- `[OPEN QUESTION]` 标出还没确认的需求

**自检（AI 执行）**
- [ ] 每条需求有没有明确的验收标准，还是只有描述？
- [ ] 还没确认的需求有没有用 `[OPEN QUESTION]` 标注？

---

## README

适用：项目、模块、仓库的入口文档。

- 30秒内让陌生人知道：这是什么、怎么用、去哪里找更多信息
- README 是地图，不是百科全书

**自检（AI 执行）**
- [ ] 陌生人读完第一段，能否说出这个项目是做什么的？
- [ ] README 里有没有可以放到其他文档里的细节？有 → 移出去，保持 README 简洁。

---

## 文档腐化防止

文档最大的问题不是写不好，是写完之后和现实脱节。

- 决策记录：状态字段必须维护，废弃的决定标"已废弃"
- 技术文档：接口变了、架构变了，文档当天更新
- README：每次项目结构大变动后过一遍

---

## 边界

- 不在读者和目的确认之前开始写（局部补充除外，但认定由用户决定）
- 不因为"看起来完整"就堆砌内容
- 同一件事只在一个地方说清楚
- 不让用户重新描述已经在对话里的信息

## 输出位置

文档放在对应业务模块的 `docs/` 目录下，路径约定见 `project-structure` skill。
