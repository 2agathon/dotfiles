## 一、命名是设计

命名不是标签，是边界声明。名字错了，边界就错了，之后所有东西都会往错的方向长。

- 用业务词，不用技术词。`InsuranceClaimValidator` 只能做一件事，`ClaimManager` 什么都能塞。
- 方法名描述意图，不描述实现。`validateClaimAmount` 告诉你发生了什么，`processData` 什么都没告诉你。
- Magic string / magic number 必须具名。`status == 2` 不知道 2 是什么，`status == ClaimStatus.UNDER_REVIEW` 意图透明。
- 同一个概念只有一个名字。发现漂移立刻指出，不等用户问。
- 避免布尔参数。调用处完全失去含义，用枚举或具名参数。

**这条规则在保护什么**
保护代码地图的准确性。命名是所有人（包括 AI）在代码库里导航的依据。名字漂移意味着地图和地形不一致，后续每一次生成都在错误地图上叠加，且不报错。

**不遵守会发生什么**
模糊命名出现三个月后，里面有校验逻辑、发送通知、写日志、调外部接口——没有人能说清楚它的边界。每次改动都要读完整个实现才敢动。

**什么情况下可以偏离**
对接外部系统时，外部接口命名不在用户控制范围内，适配层可以保留外部命名，但内部域对象必须重新命名。

**偏离时记录**
```
// DEVIATION: 保留外部字段名 insured_amt 用于序列化对接，内部使用 insuredAmount
```

**自检**
- [ ] 这个名字不看实现，能否说清楚它做了什么？
- [ ] 这个概念在其他地方有没有用不同的名字表达？

## 二、边界是架构

文件臃肿不是代码量的问题，是边界模糊的症状。

- 领域逻辑只在领域层。校验、计算、规则判断不属于入口层，也不属于数据访问层。
- 外部契约对象和内部状态对象不混用。两者变化原因不同，混用会让外部变化污染内部逻辑。
- IO 和外部依赖必须抽象成接口。写死就是把业务逻辑绑死在实现上。
- 不用无类型容器（Map / dict / any）作为返回类型。定义明确的类型，让工具帮你守边界。
- 目录结构反映业务边界，不是技术分层的刻板复制。

**这条规则在保护什么**
保护变更的可控性。边界清楚时，改一个模块不会意外影响另一个。边界模糊时，每次改动都是在一团耦合里猜哪根线能动。

**不遵守会发生什么**
入口层出现业务校验逻辑，三个月后产品改规则，用户找不到所有需要改的地方。外部契约和内部对象混用，外部接口字段变化直接污染内部计算逻辑。返回无类型容器，调用方不知道里面有什么，字段名拼错只在运行时发现。

**什么情况下可以偏离**
原型阶段、需求未稳定时，可以暂时用无类型容器或跨层引用，但必须标注并在进入迭代前处理。

**偏离时记录**
```
// DEVIATION: 暂时返回 dict，需求未稳定，进迭代前定义具名类型替换
```

**自检**
- [ ] 这个对象会被超过一层直接使用吗？
- [ ] 这个函数 / 方法的返回类型有明确定义，还是无类型容器？

## 三、接口契约

接口是边界的对外声明。入口不守，边界就是摆设。

- 不信任外部输入。所有入参在入口层做校验，不在业务逻辑里散落判断。
- 出参不泄露内部状态。返回值过滤掉内部字段、存储结构、实现细节。
- 错误码设计：稳定的 `code`（英文大写下划线）+ 可操作的 `message`（告诉调用方下一步怎么做）。不用"failed / unknown error"。
- 权限校验统一在入口层，不在业务逻辑里重复。
- 外部接口变更必须向后兼容，或者显式版本化。

**这条规则在保护什么**
保护系统边界的完整性。入口不校验，脏数据进入业务层，排查成本极高。出参泄露内部结构，调用方依赖了不该依赖的字段，之后改不动。

**不遵守会发生什么**
上游传了非法字段被写进存储。错误信息暴露了内部查询语句，安全审计直接判定漏洞。权限校验在业务层，加一个新入口就漏掉一个。调用方依赖了内部字段，重构时接口不敢动。

**什么情况下可以偏离**
内部模块间调用，调用方和被调用方在同一个边界内，可以适当简化校验。

**偏离时记录**
```
// DEVIATION: 内部模块调用，入参校验简化，调用方已保证数据合法性
```

**自检**
- [ ] 这个入口有没有在业务逻辑执行前完成入参校验？
- [ ] 这个返回值有没有暴露内部字段、存储结构或实现细节？

## 四、可见性是质量

代码的行为、决策和不确定性，应该对人可见。可见性不是注释密度，是关键信息在正确的层级出现。

### 注释按层级

不同层级的注释解决不同问题。判断标准只有一个：**这个层级的信息，读者不看实现能否自己得出？得不出就必须写。**

**边界声明层**（模块 / 包 / 命名空间）
说这组东西整体做什么、不做什么、对外暴露什么。
读者打开这个模块，30秒内能否说出它的职责边界——说不出就必须写。

**文件层**
说这个文件在业务里扮演什么角色、它不覆盖什么。
文件名已经说清楚时不需要写。文件职责不透明或承担了多个职责时必须写。

**类型定义层**（类 / 结构体 / 接口 / 枚举）
说这个类型的职责边界：只做什么，不做什么。
名字已经说清楚时不需要写。名字模糊或职责容易被误解时必须写。

**函数层**
意图清晰的简单函数不需要注释。以下情况必须写：
- 参数业务含义不透明（合法值、约束、业务语义）
- 有前置条件或副作用
- 非直觉的实现方式（说 why）
- 可能触发的异常及触发条件
- 返回值语义不透明

**成员层**（字段 / 全局变量 / 常量）
业务语义不透明的必须说明含义。
Magic value 每个值代表什么必须标注。
名字已经透明时不需要写。

**行内层**
只在 why 不明显时加。说决策原因、临时妥协、非直觉的处理方式。
what 清晰时不加行内注释——那是在重复代码本身。

### 日志

判断标准：**出了问题，这个点的信息能否帮助还原现场？能 → 必须打。不能 → 不打。**

必须打日志的点：
- 入口（请求进来时）
- 出口（返回结果时）
- 关键业务决策点（状态变更、规则判断结果）
- 外部调用（第三方接口、消息队列、存储）

不打日志的点：
- 循环体内的每次迭代
- 频繁调用的工具方法
- 能从入口/出口日志推断出来的中间步骤

每条日志必须包含：`request_id` 或业务标识、操作类型、关键业务参数、结果或错误原因。
敏感字段进日志之前必须脱敏。

### 关键决策留痕
```
// FIXME: 暂时跳过签名校验，联调期间接口未稳定，上线前必须恢复
// DEVIATION: 同步调用替代消息队列，当前流量不足以支撑引入 MQ 的成本
```

不确定的假设标注 `[ASSUMPTION]`，未解决的问题标注 `[OPEN QUESTION]`。

**这条规则在保护什么**
保护排查和维护的效率。对人不可见的决策，会在几个月后变成"这里为什么这样写"的无解之谜。

**不遵守会发生什么**
函数有多个参数，调用处不知道传什么。字段值是 2，不知道 2 代表什么。日志没有上下文，知道出错了但不知道是哪个请求触发的。临时妥协没有标注，半年后没人知道能不能动。

**什么情况下可以偏离**
注释密度根据代码自解释程度调整。名字和结构已经足够清晰时不需要注释，但参数业务含义不透明时没有例外。

**偏离时记录**
不适用——可见性本身就是记录偏离的手段。

**自检**
- [ ] 这个函数的参数，调用方不看实现能否知道传什么、有什么约束？
- [ ] 这段逻辑三个月后回来看，能否不读实现就知道为什么这样写？

## 五、错误处理

异常不是意外，是业务流程的一部分。处理方式决定了系统的可观测性和可维护性。

- 业务异常和技术异常分开处理。业务异常（参数非法、状态不符）有明确语义；技术异常（超时、网络异常）是运行时故障。两者对外暴露的方式不同。
- **在能处理的层处理，不能处理的往上抛。** 不在每一层都 catch，不在错误的层消化不该消化的异常。业务层处理业务异常，入口层统一格式化对外响应，基础设施层处理技术异常。
- 异常不能被吞。捕获了不处理，原始信息丢失，排查时无从下手。
- 往上抛时必须保留原始异常信息。
- 对外错误信息不暴露内部实现细节。内部查询语句、字段名、堆栈不出现在对外响应里。
- 错误必须可操作。告诉调用方发生了什么、下一步怎么做，不是"系统错误"。

**这条规则在保护什么**
保护排查效率和系统可信度。异常被吞，线上无从排查。对外暴露内部细节，安全漏洞和信任损失。错误信息不可操作，调用方无法处理。

**不遵守会发生什么**
线上报错没有堆栈，只能靠猜。对外返回了内部查询语句，调用方把它当契约依赖，之后改存储结构接口就炸。错误信息是"unknown error"，调用方不知道该重试还是修参数。在每一层都 catch，原始异常信息层层包装后丢失。

**什么情况下可以偏离**
顶层统一异常处理器可以捕获所有未处理异常并统一格式化，这是合理的集中处理，不是吞异常。

**自检**
- [ ] 这里捕获的异常有没有被静默处理或原始信息丢失？
- [ ] 这个对外错误响应有没有暴露内部实现细节？
- [ ] 这个异常是在正确的层处理的，还是在不该处理它的地方 catch 了？

## 六、可测试性设计

> 注意：这条和"测试本身"是两件事。可测试性设计在写代码时执行，测试本身的时机在项目 AGENTS.md 里定义。

业务逻辑和 IO 分离，才能单独验证。这不是测试的要求，是设计的要求。

- 核心计算和规则判断不依赖外部存储、网络、文件系统。
- 外部依赖通过接口注入，不在业务逻辑里直接实例化。
- 一个函数 / 方法只做一件事，副作用显式声明。

**这条规则在保护什么**
保护核心逻辑的可验证性。业务逻辑和 IO 耦合时，验证逻辑正确性必须启动整个系统，成本极高，实际上不会验证，错误在生产环境才暴露。

**不遵守会发生什么**
核心计算逻辑直接访问外部存储，改一个业务规则需要造数据、起服务、等响应，验证一次成本极高。Vibe Coding 场景下 AI 生成的逻辑无法快速验证，只能靠跑通整个流程发现问题。

**什么情况下可以偏离**
脚本类、一次性工具类代码，生命周期短，可测试性设计成本高于收益。

**偏离时记录**
```
// DEVIATION: 一次性数据迁移脚本，不做 IO 分离
```

**自检**
- [ ] 这段业务逻辑能否在不启动外部服务的情况下验证？
- [ ] 外部依赖是通过参数或接口传入的，还是在函数内部直接创建的？

## 七、AI Vibe Coding 特有规范

- 生成任何文件之前，先对齐目录和边界（触发 `project-structure` skill）。
- 发现命名漂移立刻指出，不等用户问。
- 关键节点停下来交还控制权，不从头做到尾。
- 规范是默认行为，偏离时用 `// DEVIATION:` 标注原因。
- 有取舍时主动说明：做了什么取舍、哪里规范没覆盖或表达不清楚。没有取舍时不说。

**这条规则在保护什么**
保护 Vibe Coding 的可控性和规范的持续演进。取舍不主动说明，用户不知道哪里需要关注，规范缺口也不会被发现。

**不遵守会发生什么**
命名漂移静默扩散，两周后同一个概念有三个名字。AI 做了规范没覆盖的取舍但没说，用户以为一切正常，缺口在下一个项目里继续复现。

**什么情况下可以偏离**
无合理偏离场景。

**自检**
- [ ] 生成这个文件之前，目录和业务域是否已经确认？
- [ ] 这次交付有没有做取舍？有的话是否已经主动说明？

## 八、医疗保险域特有约束

- 敏感字段（身份证号、银行卡号、手机号、诊断信息）进日志之前必须脱敏。
- 外部接口入参校验和出参过滤在这个域里没有例外——见三、接口契约。
- 权限校验统一在入口层，在这个域里没有例外——见三、接口契约。

**这条规则在保护什么**
保护合规边界。医疗保险数据属于敏感个人信息，一次泄漏是法律风险，不是代码质量问题。

**不遵守会发生什么**
身份证号出现在日志里，合规审计直接判定违规。权限校验散落在业务层，加一个新入口就漏掉一个。

**什么情况下可以偏离**
无合理偏离场景。

**自检**
- [ ] 这里打印的字段里有没有身份证号、手机号、银行卡号、诊断信息？
- [ ] 这个入口的权限校验是在入口层做的，还是在业务逻辑里散落的？
