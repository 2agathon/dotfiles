## 一、命名是设计

命名不是标签，是边界声明。名字错了，边界就错了，之后所有东西都会往错的方向长。

- 用业务词，不用技术词。`InsuranceClaimValidator` 只能做一件事，`ClaimManager` 什么都能塞。
- 方法名描述意图，不描述实现。`validateClaimAmount` 告诉你发生了什么，`processData` 什么都没告诉你。
- Magic string / magic number 必须具名。`status == 2` 不知道 2 是什么，`status == ClaimStatus.UNDER_REVIEW` 意图透明。
- 同一个概念只有一个名字。发现漂移立刻统一，不要让它蔓延。
- 避免布尔参数。`process(true, false)` 在调用处完全失去含义，用枚举或具名参数。

**这条规则在保护什么**
保护代码地图的准确性。命名是所有人（包括 AI）在代码库里导航的依据。名字漂移意味着地图和地形不一致，后续每一次生成都在错误地图上叠加，且不报错。

**不遵守会发生什么**
`ClaimManager` 出现三个月后，里面有校验逻辑、发送通知、写日志、调外部接口——没有人能说清楚它的边界。每次改动都要读完整个类才敢动。新人接手时不知道从哪里开始。

**什么情况下可以偏离**
对接外部系统时，外部接口命名不在你控制范围内，适配层可以保留外部命名，但内部域对象必须重新命名。

**偏离时记录**
```
// DEVIATION: 保留外部字段名 insured_amt 用于序列化对接，内部使用 insuredAmount
```

**自检**
- [ ] 这个名字不看实现，能否说清楚它做了什么？
- [ ] 这个概念在其他地方有没有用不同的名字表达？

## 二、边界是架构

文件臃肿不是代码量的问题，是边界模糊的症状。

- 领域逻辑只在领域层。校验、计算、规则判断不属于 Controller，也不属于 DAO。
- DTO 和领域对象不混用。DTO 是外部契约，领域对象是内部状态，两者变化原因不同。
- IO 和外部依赖必须抽象成接口。写死就是把业务逻辑绑死在实现上。
- 不用 Map / dict 作为返回类型。定义明确的类型，让编译器和 IDE 帮你守边界。
- 目录结构反映业务边界，不是技术分层的刻板复制。

**这条规则在保护什么**
保护变更的可控性。边界清楚时，改一个模块不会意外影响另一个。边界模糊时，每次改动都是在一团耦合里猜哪根线能动。

**不遵守会发生什么**
Controller 里出现业务校验逻辑，三个月后产品改规则，你找不到所有需要改的地方。DTO 和领域对象混用，外部接口字段变化直接污染内部计算逻辑，数据库表加字段导致前端接口报错。返回裸 dict / Map，调用方不知道里面有什么，IDE 无法提示，字段名拼错只在运行时发现。

**什么情况下可以偏离**
原型阶段、需求未稳定时，可以暂时用 dict 或跨层引用，但必须标注并在进入迭代前处理。

**偏离时记录**
```
// DEVIATION: 暂时返回 dict，需求未稳定，进迭代前定义 PolicySummary 类替换
```

**自检**
- [ ] 这个对象会被超过一层直接使用吗？
- [ ] 这个函数 / 方法的返回类型有明确定义，还是 dict / Map / any？

## 三、可见性是质量

代码的行为、决策和不确定性，应该对人可见。

- 注释说 why，不说 what：
```
  // ✗ 判断状态是否为2
  // ✓ 状态2表示"人工复核中"，此时不允许用户撤销申请（防止审核资源浪费）
```
- 关键决策必须留痕。临时妥协必须注释原因和计划：
```
  // FIXME: 暂时跳过签名校验，联调期间接口未稳定，上线前必须恢复
  // DEVIATION: 同步调用替代消息队列，当前流量不足以支撑引入 MQ 的成本
```
- 日志要有上下文，出了问题只看日志就能还原当时发生了什么。
- 敏感字段进日志之前必须脱敏。
- 异常不能被吞，原始信息不能丢。包一层再抛时必须保留原始异常。
- 不确定的假设标注 `[ASSUMPTION]`，未解决的问题标注 `[OPEN QUESTION]`。

**这条规则在保护什么**
保护排查和维护的效率。出了问题能不能快速还原现场，完全取决于代码写的时候留没留痕迹。对人不可见的决策，会在几个月后变成"这里为什么这样写"的无解之谜。

**不遵守会发生什么**
异常被吞，线上报错没有堆栈，只能靠猜。日志没有上下文，知道出错了但不知道是哪个请求、哪个数据触发的。敏感字段进日志，合规审计时变成事故。临时妥协没有标注，半年后没人知道能不能动。

**什么情况下可以偏离**
注释密度可以根据代码自解释程度调整。逻辑足够清晰时不需要强制加注释，但 why 不明显时没有例外。

**偏离时记录**
不适用——可见性本身就是记录偏离的手段。

**自检**
- [ ] 这段逻辑三个月后回来看，能否不读实现就知道为什么这样写？
- [ ] 这里的异常有没有被静默处理或原始信息丢失？

## 四、可测试性设计

> 注意：这条和"测试本身"是两件事。可测试性设计在写代码时执行，测试本身的时机在项目 AGENTS.md 里定义。

业务逻辑和 IO 分离，才能单独测试。这不是测试的要求，是设计的要求。

- 核心计算和规则判断不依赖数据库、HTTP、文件系统。
- 外部依赖通过接口注入，不在业务逻辑里直接实例化。
- 一个函数 / 方法只做一件事，副作用显式声明。

**这条规则在保护什么**
保护核心逻辑的可验证性。业务逻辑和 IO 耦合时，验证逻辑正确性必须启动整个系统，成本极高，所以实际上不会验证，错误会在生产环境才暴露。

**不遵守会发生什么**
保费计算逻辑直接查数据库，改一个费率规则需要造数据、起服务、等响应，验证一次十分钟。Vibe Coding 场景下 AI 生成的逻辑无法快速验证，只能靠跑通整个流程来发现问题，调试成本翻倍。

**什么情况下可以偏离**
脚本类、一次性工具类代码，生命周期短，可测试性设计成本高于收益。

**偏离时记录**
```
// DEVIATION: 一次性数据迁移脚本，不做 IO 分离
```

**自检**
- [ ] 这段业务逻辑能否在不启动数据库 / 外部服务的情况下验证？
- [ ] 外部依赖是通过参数或接口传入的，还是在函数内部直接创建的？

## 五、AI Vibe Coding 特有规范

- 生成任何文件之前，先对齐目录和边界（触发 `project-structure` skill）。
- 发现命名漂移立刻指出，不等用户问。
- 关键节点停下来交还控制权，不从头做到尾。
- 规范是默认行为，偏离时用 `// DEVIATION:` 标注原因。

**这条规则在保护什么**
保护 Vibe Coding 的可控性。AI 生成速度快，但方向一旦偏了修正成本极高。提前对齐比事后重构便宜得多。

**不遵守会发生什么**
文件路径生成错误，后续所有 import 都要改。命名漂移静默扩散，两周后同一个概念有三个名字。AI 从头做到尾交付一个大 PR，里面有五个需要讨论的设计决策但已经全部实现完了。

**什么情况下可以偏离**
无合理偏离场景。

**自检**
- [ ] 生成这个文件之前，目录和业务域是否已经确认？
- [ ] 这次生成有没有跨过一个需要用户决策的节点？

## 六、医疗保险域特有约束

- 敏感字段（身份证号、银行卡号、手机号、诊断信息）进日志之前必须脱敏。
- 权限校验统一在入口层，不在业务逻辑里散落判断。
- 外部接口必须做入参校验和出参过滤，不信任外部数据。

**这条规则在保护什么**
保护合规边界。医疗保险数据属于敏感个人信息，一次泄漏是法律风险，不是代码质量问题。

**不遵守会发生什么**
身份证号出现在日志里，合规审计直接判定违规。权限校验散落在业务层，加一个新入口就漏掉一个。外部接口不过滤，上游传了多余字段被写进数据库或转发给下游。

**什么情况下可以偏离**
无合理偏离场景。

**自检**
- [ ] 这里打印的字段里有没有身份证号、手机号、银行卡号、诊断信息？
- [ ] 这个入口的权限校验是在入口层做的，还是在业务逻辑里做的？
