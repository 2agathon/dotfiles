## 这个 skill 的边界

**适用：** 任何写代码、命名、设计接口的场景。
**不适用：** 脚本类一次性工具、原型阶段（具体偏离条件见各规则）。
**遇到不适用的场景：** 说明原因，用 `// DEVIATION:` 标注，继续推进。

---

## 强制停止点

以下情况 AI 必须停下来，不能用自检或 ASSUMPTION 绕过，必须等用户确认：

- 发现命名漂移（同一概念有多个名字）——列出所有漂移点，等用户决定统一哪个
- 发现文件边界模糊需要拆分——列出拆分方案，等用户确认后再动
- 接口变更影响已有调用方——列出影响范围，等用户确认兼容策略
- 错误处理层级有歧义——说明歧义，等用户决定在哪层处理

## 一、命名是设计

命名不是标签，是边界声明。名字错了，边界就错了，之后所有东西都会往错的方向长。

- 用业务词，不用技术词。`PaymentAmountValidator` 只能做一件事，`PaymentManager` 什么都能塞。
- 方法名描述意图，不描述实现。`validateOrderTotal` 告诉你发生了什么，`processData` 什么都没告诉你。
- Magic string / magic number 必须具名。`status == 2` 不知道 2 是什么，`status == OrderStatus.PENDING_REVIEW` 意图透明。
- 同一个概念只有一个名字。**发现漂移立刻停下来，列出所有漂移点，等用户决定——不能自己选一个然后继续。**
- 避免布尔参数。调用处完全失去含义，用枚举或具名参数。

**这条规则在保护什么**
保护代码地图的准确性。命名是所有人（包括 AI）在代码库里导航的依据。名字漂移意味着地图和地形不一致，后续每一次生成都在错误地图上叠加，且不报错。

**不遵守会发生什么**
模糊命名出现三个月后，里面有校验逻辑、发送通知、写日志、调外部接口——没有人能说清楚它的边界。每次改动都要读完整个实现才敢动。

**什么情况下可以偏离**
对接外部系统时，外部接口命名不在用户控制范围内，适配层可以保留外部命名，但内部域对象必须重新命名。

**偏离时记录**
```
// DEVIATION: 保留外部字段名 ext_order_amt 用于序列化对接，内部使用 orderAmount
```

**自检（AI 执行，发现违反时停下来，不能自己选名字继续）**
- [ ] 这个名字不看实现，能否说清楚它做了什么？
- [ ] 这个概念在其他地方有没有用不同的名字表达？有 → 停下来，等用户决定。

## 二、边界是架构

文件臃肿不是代码量的问题，是边界模糊的症状。

- 领域逻辑只在领域层。校验、计算、规则判断不属于入口层，也不属于数据访问层。
- 外部契约对象和内部状态对象不混用。两者变化原因不同，混用会让外部变化污染内部逻辑。
- IO 和外部依赖必须抽象成接口。写死就是把业务逻辑绑死在实现上。
- 不用无类型容器（Map / dict / any）作为返回类型。
- 目录结构反映业务边界，不是技术分层的刻板复制。

**这条规则在保护什么**
保护变更的可控性。边界清楚时，改一个模块不会意外影响另一个。边界模糊时，每次改动都是在一团耦合里猜哪根线能动。

**不遵守会发生什么**
入口层出现业务校验逻辑，三个月后产品改规则，找不到所有需要改的地方。返回无类型容器，字段名拼错只在运行时发现。

**什么情况下可以偏离**
原型阶段、需求未稳定时，可以暂时用无类型容器或跨层引用，但必须标注并在进入迭代前处理。

**偏离时记录**
```
// DEVIATION: 暂时返回 dict，需求未稳定，进迭代前定义具名类型替换
```

**自检（AI 执行）**
- [ ] 这个对象会被超过一层直接使用吗？
- [ ] 返回类型有明确定义，还是无类型容器？

## 三、接口契约

接口是边界的对外声明。入口不守，边界就是摆设。

- 不信任外部输入。所有入参在入口层做校验，不在业务逻辑里散落判断。
- 出参不泄露内部状态。返回值过滤掉内部字段、存储结构、实现细节。
- 错误码设计：稳定的 `code`（英文大写下划线）+ 可操作的 `message`（告诉调用方下一步怎么做）。
- 权限校验统一在入口层，不在业务逻辑里重复。
- 外部接口变更必须向后兼容，或者显式版本化。

**接口变更的强制停止点：** 任何会影响已有调用方的接口变更，必须列出影响范围等用户确认，不能自己决定兼容策略。

**什么情况下可以偏离**
内部模块间调用，调用方和被调用方在同一边界内，可以适当简化校验。

**偏离时记录**
```
// DEVIATION: 内部模块调用，入参校验简化，调用方已保证数据合法性
```

**自检（AI 执行）**
- [ ] 入口有没有在业务逻辑执行前完成入参校验？
- [ ] 返回值有没有暴露内部字段或实现细节？

## 四、可见性是质量

代码的行为、决策和不确定性，应该对人可见。判断标准：**读者不看实现能否自己得出这个信息？得不出就必须写。**

### 注释按层级

| 层级                  | 什么时候必须写                                             |
| --------------------- | ---------------------------------------------------------- |
| 边界声明层（模块/包） | 30秒内说不出职责边界                                       |
| 文件层                | 文件职责不透明或承担多个职责                               |
| 类型定义层            | 名字模糊或职责容易被误解                                   |
| 函数层                | 参数含义不透明、有前置条件、有副作用、非直觉实现、可能异常 |
| 成员层                | 业务语义不透明、magic value                                |
| 行内层                | why 不明显时，说决策原因或临时妥协                         |

### 日志

判断标准：**出了问题，这个点的信息能否帮助还原现场？能 → 必须打。**

必须打：入口、出口、关键业务决策点、外部调用。
不打：循环体每次迭代、频繁调用的工具方法、能从入口/出口推断的中间步骤。
每条日志必须包含：业务标识、操作类型、关键参数、结果或错误原因。
敏感字段进日志前必须脱敏。

### 关键决策留痕
```
// FIXME: 暂时跳过签名校验，联调期间接口未稳定，上线前必须恢复
// DEVIATION: 同步调用替代消息队列，当前流量不足以支撑引入 MQ 的成本
```

**这条规则在保护什么**
保护排查和维护的效率。对人不可见的决策，会在几个月后变成无解之谜。

**不遵守会发生什么**
函数有多个参数，调用处不知道传什么。日志没有上下文，知道出错了但不知道是哪个请求触发的。临时妥协没有标注，半年后没人知道能不能动。

**什么情况下可以偏离**
名字和结构已经足够清晰时不需要注释，但参数业务含义不透明时没有例外。

**自检（AI 执行）**
- [ ] 函数参数，调用方不看实现能否知道传什么、有什么约束？
- [ ] 这段逻辑三个月后回来看，不读实现能否知道为什么这样写？

## 五、错误处理

- 业务异常和技术异常分开处理。
- **在能处理的层处理，不能处理的往上抛。**
- 异常不能被吞。
- 往上抛时必须保留原始异常信息。
- 对外错误信息不暴露内部实现细节。
- 错误必须可操作，告诉调用方下一步怎么做。

**错误处理层级有歧义时：** 停下来说明歧义，等用户决定在哪层处理，不能自己选择后继续。

**不遵守会发生什么**
线上报错没有堆栈，只能靠猜。对外返回了内部查询语句，调用方把它当契约依赖，之后改存储结构接口就炸。

**什么情况下可以偏离**
顶层统一异常处理器捕获所有未处理异常并统一格式化，这是合理的集中处理，不是吞异常。

**自检（AI 执行）**
- [ ] 捕获的异常有没有被静默处理或原始信息丢失？
- [ ] 这个异常是在正确的层处理的，还是在不该处理它的地方 catch 了？

## 六、可测试性设计

> 注意：这条和"测试本身"是两件事。可测试性设计在写代码时执行，测试本身的时机在项目 AGENTS.md 里定义。

- 核心计算和规则判断不依赖外部存储、网络、文件系统。
- 外部依赖通过接口注入，不在业务逻辑里直接实例化。
- 一个函数 / 方法只做一件事，副作用显式声明。

**什么情况下可以偏离**
脚本类、一次性工具类代码，生命周期短，可测试性设计成本高于收益。

**偏离时记录**
```
// DEVIATION: 一次性数据迁移脚本，不做 IO 分离
```

**自检（AI 执行）**
- [ ] 这段业务逻辑能否在不启动外部服务的情况下验证？
- [ ] 外部依赖是通过参数或接口传入的，还是在函数内部直接创建的？

## 七、AI Vibe Coding 特有规范

- 生成任何文件之前，先对齐目录和边界（触发 `project-structure` skill）。
- 发现命名漂移立刻停下来——列出所有漂移点，等用户决定，不能自己选一个继续。
- 关键节点停下来交还控制权，不从头做到尾。
- 有取舍时主动说明：做了什么取舍、哪里规范没覆盖或表达不清楚。没有取舍时不说。

**什么情况下可以偏离**
无合理偏离场景。

**自检（AI 执行）**
- [ ] 生成文件之前，目录和业务域是否已经确认？
- [ ] 这次交付有没有做取舍？有的话是否已经主动说明？
